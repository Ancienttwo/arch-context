## P3 Decision

FG4-01 keeps scope and lifecycle rules in contracts instead of in the runner package or control-plane storage. This preserves one reusable invariant for GitHub-hosted, self-hosted, and future CI adapters. The schema remains backward compatible with the PRD example: an identity without `scope` is interpreted as repository scope. Explicit organization scope is added as an optional contract extension, but the domain still rejects ambiguous states such as organization scope with repository IDs, repository scope with no repository IDs, invalid workflow refs, revoked identities without `revokedAt`, and rotating identities without `rotatedAt`.

FG4-02 keeps lifecycle execution in the control-plane but still uses the shared RunnerIdentity domain for every state. Rotation creates a new active RunnerIdentity rather than overwriting the old key; that preserves auditability and lets the overlap window be explicit. The current in-memory store is deliberate for this slice because FG5 owns migrations and durable transaction semantics. At 10x scale, the first pressure point is indexing runner identities by installation/scope/public key and moving the overlap window into durable storage; FG4-02 only freezes lifecycle behavior and metadata-only surfaces.

FG4-03 keeps authorization proof as a typed input to the control-plane rather than teaching the lifecycle code how to call GitHub. This preserves the existing typed GitHub port boundary: future API code can obtain the proof from GitHub App/OAuth readback, while the lifecycle path only consumes a minimal proof and records an audit event. At 10x scale, the first pressure point is durable audit storage and idempotent write keys; those belong to FG5, while this slice proves no public Runner Key lifecycle operation can bypass admin proof.

FG4-04 keeps organization run metadata inside `execution` instead of adding a new top-level attestation object. The invariant is that execution identity, signing key, workflow ref, and GitHub run coordinates are one signed unit. This preserves developer attestation compatibility and prevents a verifier from accepting a generic organization trust flag without knowing which runner/workflow/run produced it. At 10x scale, the first pressure point is verifier lookup against durable RunnerIdentity state and GitHub run readback; those are FG4-13/14 and FG5 work, while this slice freezes the signed contract shape.

FG4-05 deliberately makes the first official Action slice a preflight and pinning gate, not a full review execution path. That keeps FG4-05 aligned with its release-target dependency and avoids duplicating checkout/verifier logic before FG4-06/10/13. At 10x scale, the first pressure point is release artifact provenance and digest publication; this slice freezes the consumer-side verification contract and leaves artifact publishing plus real workflow execution for the following runner tasks.

FG4-06 reuses local-runtime git-adapter semantics rather than inventing a second checkout validator in runner code. The invariant is that the organization runner must execute on the exact PR head and tree bound to the Challenge, not a merge ref, stale checkout, branch tip, or dirty working copy. At 10x scale, the first pressure point is Challenge leasing/resolution in the Action, which remains FG4-15; this slice freezes the verification contract and proves it against real temporary Git repositories.

FG4-07 treats job permissions as a runner security contract instead of relying on prose in the workflow template. The invariant is that the customer workflow can read the checked-out PR and metadata but cannot write repository contents or request a broad GitHub token. At 10x scale, the pressure point is fork/security policy branching, which remains FG4-08/09; this slice freezes the default safe baseline.

FG4-08 keeps trigger safety separate from permissions because `pull_request_target` can expose secrets even when the permission block looks modest. The invariant is that untrusted PR code never runs on a privileged target event by default. At 10x scale, the next pressure point is the fork PR outcome and no-secret fallback, which is FG4-09.

FG4-09 keeps fork policy separate from checkout validation because fork safety is an event/secret boundary, not a Git tree boundary. The invariant is that the default Organization Runner path cannot accidentally mint a Challenge or use a signing secret for fork PR code. At 10x scale, the first pressure point is proving this against a real public fork workflow and GitHub-required Check behavior; this slice freezes the local policy and Check semantics, while FG4-EG5 still owns public fork adversarial staging readback.

FG4-10 keeps no-LLM execution in the runner package instead of adding provider flags to the review engine. The invariant is that deterministic pass/fail comes from `completeTaskGate`; the model slot is represented by a fixed no-provider digest for signing stability, not by a provider credential. At 10x scale, the pressure point is keeping future LLM advisory output from influencing this digest or conclusion, which is exactly the next FG4-11 slice.

FG4-11 treats LLM Advisory as a local optional artifact rather than a Check publisher or Attestation input. The invariant is that advisory text can explain or suggest repair steps, but it cannot carry or override required-check conclusion state. At 10x scale, the first pressure point is making sure future Cloud storage and GitHub comments cannot accidentally persist advisory text as policy evidence; this slice blocks conclusion-shaped fields and leaves any advisory presentation surface to later product work.

FG4-12 keeps private key custody behind a customer Secret Store ref instead of passing a `KeyObject` through the public runner API. The invariant is that workflow inputs, logs, artifacts, caches, review digests, and upload payloads can carry refs and metadata, not PEM or file paths. At 10x scale, the first pressure point is inspecting real GitHub Actions artifacts/caches/logs, which remains FG4-EG8; this slice closes the runner API and local projection boundary.

FG4-13 keeps the runner output on Attestation v2 instead of widening the legacy local-attestation extension object. The invariant is that Organization required checks must verify one canonical v2 payload containing runner identity, workflow run identity, runtime build provenance, Challenge nonce, and review digest. At 10x scale, the first pressure point is verifier lookup against active RunnerIdentity state and workflow scope, which remains FG4-14; this slice makes the signed payload carry the required facts.

FG4-14 keeps RunnerIdentity enforcement inside the attestation verifier rather than only in control-plane call sites. The invariant is that every Organization Attestation v2 required check crosses the same identity/scope/key-status gate, whether submitted synchronously or through future delivery paths. At 10x scale, the first pressure point is durable lookup and lease ownership; those remain FG4-15/FG5, while this slice freezes the in-memory verifier contract and reason-code behavior.

FG4-15 keeps lease heartbeat/retry in control-plane rather than the Action entrypoint. The invariant is that runner concurrency and timeout behavior is decided by the Challenge owner state, not by local workflow timing alone. At 10x scale, the first pressure point is durable compare-and-swap around lease writes; this slice keeps the API pure and deterministic so FG5 can add persistence without changing caller semantics.

FG4-16 keeps Organization Runner Check publication in a dedicated GitHub App method rather than parameterizing the Developer Review publisher. The invariant is user-visible context isolation: a passing Developer Attestation must never turn the Organization Runner required check green, and a valid Organization Attestation must not render under the Developer Review header. At 10x scale, the first pressure point is delivery idempotency and retry around real CheckRun updates, which remains FG5; this slice freezes the trust-context boundary and publication summary contract.

FG4-17 keeps the GitHub-hosted workflow template in `docs/examples` rather than enabling it in this repository's active `.github/workflows`. The invariant is that customers get a copyable, tested template without accidentally running Organization Runner jobs in the product repo before runtime artifact publication and real Challenge auto-resolution are ready. At 10x scale, the first pressure point is replacing the template placeholders with durable Challenge resolver outputs; that belongs to the later real workflow readback slices, while this slice freezes the secure workflow shape.

FG4-18 keeps self-hosted hardening as a security guide plus regression test rather than an installer script. The invariant is that ArchContext must not pretend to own the customer's host isolation, firewall, or Secret Store, but it must state the minimum boundary required for an Organization Attestation to be meaningful. At 10x scale, the first pressure point is validating real artifacts, caches, logs, and outbound traces from customer runners; that remains FG4-EG8, while this slice freezes the documented baseline.

FG4-19 keeps the reusable workflow separate from the direct GitHub-hosted template so customers can choose direct vendoring or central workflow reuse. The invariant is release pinning: the caller must be able to bind the reusable workflow contents by immutable commit SHA, while the reusable workflow still preserves the same no-Contents-write, exact-head, runtime-pinned runner contract. At 10x scale, the first pressure point is publishing the reusable workflow from a stable release repository and readbacking a real call; that remains FG4-EG1 and release work, while this slice freezes the file contract and caller shape.

FG4-20 keeps the expected-source proof as a temporary staging ruleset rather than a persistent branch protection change. The invariant is that GitHub recognizes `ArchContext / Organization Runner` as the required context produced by the ArchContext App integration, while the test repository is returned to its prior state after readback. At 10x scale, the first pressure point is durable ruleset management and retry around protected branch policies; that remains FG5/FG6, while this slice proves the staging API behavior and cleanup.

FG4-21 keeps revoke and unregister as irreversible key lifecycle transitions instead of adding a recoverable `active` transition. The invariant is that a revoked or unregistered key can never make an Organization required check green again; recovery means registering replacement public-key metadata and updating the customer's Secret Store. At 10x scale, the first pressure point is durable storage for termination kind, audit events, and idempotent replacement registration; FG4-21 freezes the caller-visible behavior in the control-plane before FG5 persistence.

FG4-EG1 deliberately uses a temporary PR instead of pushing the dirty FG4 work to the standing staging PR #2. The invariant under test is the live GitHub-hosted execution path and App-owned Organization Check source, not branch lifecycle management. The temporary PR keeps the production branch and existing staging PR unchanged while still proving exact-head checkout, no-LLM deterministic proof generation, metadata-only artifact upload, App CheckRun success, and cleanup. At 10x scale, the pressure point is replacing this harness with a durable customer workflow and Challenge resolver; that remains later release/persistence work.

FG4-EG3 keeps policy proof at the GitHub ruleset/Check context boundary rather than relying only on local `satisfiesRequiredTrust` unit tests. The invariant is that a green Developer Check never satisfies a protected branch rule requiring the Organization Runner context, even when both Checks are produced by the same App installation on the same head. The evidence reuses a temporary ruleset instead of changing persistent branch protection, preserving staging repository state while proving the protected-branch semantics.

FG4-EG4 keeps required-check submission active-only instead of accepting rotating previous keys during overlap. The invariant is that rotation grace can help admin/preflight migration, but the key that turns an Organization required check green must be the current active RunnerIdentity. At 10x scale, the first pressure point is durable rotation-window storage and transactionally looking up active RunnerIdentity/key status during submit; FG4-EG4 freezes the in-memory caller-visible lifecycle semantics before FG5 persistence.

FG4-EG6 keeps supply-chain protection split across two boundaries: Action preflight rejects obviously replaced runtime inputs before execution, and Attestation v2 signature verification rejects post-signing mutation of runtime provenance or workflow run metadata. The invariant is that required-check conclusion is bound to the exact pinned runtime artifact digest and run identity. At 10x scale, the first pressure point is publishing immutable release artifacts and expected digest discovery; this slice proves the consumer and verifier contracts before release automation hardens that distribution path.

FG4-EG7 keeps advisory text outside the required-check conclusion path instead of adding an allowlist of advisory providers that can override gate output. The invariant is that the Check conclusion comes from deterministic review evidence signed into Attestation v2; LLM Advisory can only add separate human-readable material. At 10x scale, the first pressure point is future comment/storage surfaces accidentally treating advisory as policy evidence; this slice keeps the runner payload and signed conclusion clean.

FG4-EG8 keeps DLP evidence as scan summaries instead of storing raw logs or artifacts in the repository. The invariant is that runner artifacts, cache output, logs, and Cloud DTOs can be inspected for code-content leakage without turning the evidence packet itself into a leakage vector. At 10x scale, the first pressure point is making this scan part of every live workflow readback; this slice proves the scanner against the existing real GitHub-hosted Organization Runner run.

## Known Limitations

FG4-01 through FG4-21, FG4-EG1, FG4-EG3, FG4-EG4, FG4-EG6, FG4-EG7, and FG4-EG8 do not persist Runner Keys/audit events durably, publish the runtime artifact, wire lease heartbeat into the real Action entrypoint, present advisory output in a product UI, or prove public fork workflows. FG4-11 covers advisory/conclusion isolation, but not future advisory sharing, comment publication, or storage policy. FG4-12 covers runner API custody and local surface rejection; FG4-EG8 covers the real EG1 GitHub-hosted artifact/log/cache/Cloud DTO scan but not a future self-hosted runner host. FG4-13 signs the v2 payload locally, FG4-14 enforces active RunnerIdentity scope/key status in the in-memory verifier, and FG4-15 freezes in-memory lease heartbeat/retry semantics; durable lookup and compare-and-swap lease writes remain FG5. FG4-16 proves the dedicated Organization Runner publication path locally and reuses existing staging context readback for distinct CheckRun evidence. FG4-17 provides a tested GitHub-hosted workflow template, but durable Challenge auto-resolution and runtime artifact publication remain later work. FG4-18 documents the self-hosted hardening baseline, but does not inspect a real self-hosted runner host or firewall. FG4-19 provides a reusable workflow and pinned caller shape, but does not yet publish it from a stable external release repository or run a live workflow_call readback. FG4-20 and FG4-EG3 verify expected-source and protected-ruleset behavior with temporary staging rulesets, but do not leave persistent branch protection enabled. FG4-21 covers in-memory unregister/revoke recovery UX and submit rejection, but not a shipped admin UI, CLI command, or durable termination record. FG4-EG1 proves the GitHub-hosted staging path through a temporary PR harness, not a permanent customer workflow installation. FG4-EG4 is a process-level E2E over lifecycle and submit semantics, not a live GitHub runner rotation. FG4-EG6 proves consumer/verifier negative behavior, but not release artifact publication or CDN immutability. FG4-EG7 proves local runner conclusion isolation, but not future advisory comment or UI presentation surfaces.
