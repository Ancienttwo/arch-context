## P3 Decision

`verify:governance` stays local and evidence-inspection based instead of mutating staging on every CI run. The invariant is that CI can block regressions in contracts, privacy checks, ledger state, and committed E3 evidence without requiring `_ops/env`, GitHub tokens, Cloudflare credentials, or Wrangler deploy access. Live staging readbacks remain explicit operator commands, while this slice freezes the repeatable command and hosted job that later FG6 gates can extend.

FG6-02 extends the existing FG1 no-cloud E2E instead of adding a second simulated product path. That preserves the invariant that release evidence must exercise the installed local product surface, not a unit-level mock. The tradeoff is that AC-01 remains local E3 evidence rather than a hosted staging mutation, which is correct because AC-01 explicitly requires no GitHub App, Cloud account, or LLM provider. At 10x scale, the first pressure point is platform variance for IPC and installed-bin path resolution; that remains FG6-08, while this slice is sufficient to freeze the no-cloud first experience.

FG6-03 composes existing FG3 E3 artifacts instead of creating another live PR mutation. The invariant is that release evidence is immutable and inspectable without `_ops/env`, while the underlying source artifacts already prove the real staging Check publication and exact-head clean worktree path. The tradeoff is that this slice does not refresh the live GitHub Check; it freezes the most recent verified staging proof. At 10x scale, stale or duplicate Check ordering remains the AC-03/FG6-04 pressure point, not this AC-02 provenance slice.

FG6-04 composes existing live PR synchronize and supersede artifacts rather than pushing another temporary commit. The invariant is that AC-03 release verification can prove new-head invalidation without consuming GitHub/App credentials during every governance verify run. The tradeoff is that this is immutable E3 evidence, not a fresh staging mutation. That is sufficient for the release gate because the source artifacts already include a real temporary PR, real CheckRun IDs, and focused Control Plane stale-submit tests; later chaos/retry behavior remains FG6-10 and FG6-EG5.

FG6-05 composes FG4 live runner evidence instead of launching another GitHub Actions run. The invariant is that release verification remains inspect-only, while the source evidence already proves a real customer-controlled GitHub-hosted runner path with no LLM provider configured. The tradeoff is that this slice freezes the verified runner run rather than refreshing it live; platform matrix and self-hosted variance remain separate FG6 slices.

FG6-06 composes existing static, dynamic, runner, and storage DLP evidence rather than replaying another full PR lifecycle. The invariant is that AC-05 release verification must remain CI-safe and inspect-only while still proving all privacy surfaces that would carry code content: GitHub API allowlist, cloud DTO/egress/tail logs, runner artifacts/logs/cache, and D1/control-plane exports. The tradeoff is that this slice freezes the last verified staging/run artifacts; permission degradation and fork/revoke adversarial refresh are covered by FG6-09, while this slice is sufficient to prove the zero code-content routing contract across static, dynamic, and storage surfaces.

FG6-07 composes no-provider deterministic sources instead of rerunning the official Review Action. The invariant is that required-check pass/fail must be stable without provider credentials and must come from deterministic runtime evidence, not Advisory or model output. The tradeoff is that this slice does not refresh the live GitHub Actions run; it freezes the last verified official action plus process-level deterministic conclusion proof, which is sufficient for AC-06 because platform matrix refresh is covered by FG6-08 and fork/permission degradation is covered by FG6-09.

FG6-08 composes matrix evidence rather than dispatching a fresh hosted workflow from the release gate. The invariant is that `verify:governance` remains token-free and local while still detecting drift in the workflow matrix, installed-bin IPC readback, and previously verified hosted artifacts. The tradeoff is that the run IDs are immutable prior executions rather than a fresh CI dispatch; this is sufficient for the support-matrix slice because the current workflow contract is checked locally and the fork/permission matrices are covered by FG6-09.

FG6-09 composes adversarial evidence rather than re-suspending the GitHub App installation or reopening another public fork PR. The invariant is that the final release gate must detect governance regressions without consuming operator credentials or mutating staging during ordinary CI. The tradeoff is that it freezes the last verified live fork/revoke/ruleset artifacts; this is sufficient for the adversarial matrix slice because each source artifact already exercised the real GitHub App, public fork namespace, temporary ruleset, and key lifecycle path, while fault injection and chaos behavior remain FG6-10.

FG6-10 composes reliability evidence instead of triggering new staging faults on every release-gate run. The invariant is that CI can reject missing Webhook, DB, Queue, GitHub API, and clock-skew resilience proof without requiring `_ops/env` or changing live state. The tradeoff is that staging GitHub API failure evidence is immutable while DB/clock-skew evidence is anchored to the FG5 gate and current regression source; this is sufficient for the chaos matrix slice because the real staging route already exercised retry/DLQ/replay and the remaining failure classes are transaction/validation contracts covered by full `bun run verify`.

FG6-11 uses a repo-local release security bundle instead of relying on an ad hoc manual scan note. The invariant is that a release cannot claim security readiness unless dependency advisories, SBOM inventory, high/critical static patterns, secret markers, and security manifest readback agree on zero unaccepted Critical/High findings. The tradeoff is that this is a deterministic staging release scan, not the external independent security review required by FG6-12; it is sufficient for FG6-11 because the artifact is repeatable, manifest-backed, and included in `verify:governance`.

FG6-12 composes existing immutable evidence instead of rerunning destructive fork/revoke/staging probes. The invariant is that the review conclusion must fail if any source inspector fails or any covered security class regresses. The tradeoff is that this is an independent release evidence review, not a production penetration test; that is sufficient for the E3 sprint task because every reviewed class is backed by prior live or process-level evidence and now has one external review disposition.

FG6-13 uses a real sibling workspace repo instead of generating an artificial large fixture. The invariant is that performance evidence should measure the actual installed product and CodeGraph path against a clean Git checkout, while avoiding writes to the source repo. The tradeoff is that `aiphabee` has 1073 tracked files rather than the PRD's 10000-file worst-case reference; this slice is sufficient for FG6-13 because it records a repeatable representative benchmark and leaves broader SLO/budget interpretation to FG6-14 and FG6-EG7.

FG6-14 adds a first-class `challenge_create_latency_ms` Control Plane metric instead of treating Challenge creation as a paper-only SLO. The invariant is that release SLOs must bind to metadata-only telemetry and immutable readback evidence without expanding the GitHub API, logging source content, or requiring `_ops/env` in aggregate CI. The tradeoff is that FG6-14 verifies release evidence samples and source coverage, not production rolling-window SLO compliance; that is sufficient for this E3 slice because production aggregation and rollout monitoring are later FG6 rollout gates.

FG6-15 fixes account deletion at the existing Control Plane state boundary rather than adding a new durable account-deletion service. The invariant is that account-scoped local control-plane state must be removed without touching the already-proven remote D1 retention and installation revoke mechanisms. The tradeoff is that account delete is verified as an in-memory control-plane probe while durable D1 retention remains the remote staging proof; this is sufficient for FG6-15 because the task asks for a deletion drill across retention, installation revoke, and account delete rather than a production account-deletion workflow.

FG6-16 composes runbook readiness from existing incident evidence instead of forcing another live GitHub outage, key revoke, or queue backlog during release-gate verification. The invariant is that operators need concrete, evidence-backed steps while the repeatable governance gate remains token-free and non-mutating. The tradeoff is that this slice validates runbook coverage and source evidence alignment, not a new production incident exercise; that is sufficient for the E3 task because the underlying Device/Runner revoke, GitHub API failure, DLQ/replay, and alert evidence already exists and is re-inspected.

FG6-17 keeps rollout switches in one shared contract and applies them at every side-effect boundary instead of scattering boolean checks in only the GitHub App. The invariant is that disabling a flag must stop new Check/Challenge/queue/publication side effects before state is written, while `requiredTrust:false` should degrade organization-required webhook handling to the already-safe Developer Check rather than silently publishing Organization Runner evidence. The tradeoff is that this E2 release slice verifies local gate behavior and immutable source coverage, not design-partner rollout telemetry; that is sufficient because the current release target is a personal-user Beta.

FG6-19 composes rollback compatibility into one release readback instead of relying on separate unit tests. The invariant is that rollback must preserve old Attestation readability as audit-only data while never upgrading low-trust or legacy evidence into a required Check pass. The tradeoff is that this is a local E3 drill, not a production rollback exercise; it is sufficient because it executes the schema rejection, legacy migration, Check Context mismatch, and Action runtime-version pinning paths that would fail first during a rollback.

The release distribution readback intentionally separates npm publication from collaboration rollout evidence. The invariant is that a personal user must be able to install a public release artifact, not rely on a source checkout, local tarball, or staging-only Worker deploy. The tradeoff is that team/cohort rollout telemetry moves to TODO, while the current slice stays focused on the individual install and no-cloud first-run.

The deferred rollout intake keeps the generated operator packet deliberately failing instead of manufacturing a green collaboration artifact. The invariant is that future design-partner or opt-in beta acceptance must come from real telemetry, with ordered phases, non-empty cohorts, security controls, and SLO metrics. The tradeoff is that current FG6 can close personal-user usability without claiming collaboration rollout health.
