## P2 Trace

The traced path is: typed pull/head metadata plus policy input issues a `ReviewChallengeV2` value with a crypto-random base64url nonce and a default 15-minute expiry, then completed runtime review metadata and digests create an `AttestationV2` value. Both paths validate installation/repository/PR/head/base/trust/time/status fields, reject private-content shaped fields, serialize through the shared contracts canonical JSON function, and derive stable digests. Attestation v2 canonical bytes project `signature.value` to an empty string while preserving `signature.algorithm`, matching the PRD signing boundary. For Device Key registration, the control-plane path receives account ID, public key ID, a public key or explicit fingerprint, and creation time; `publicKeyFingerprint` derives a `sha256:` fingerprint from the public SPKI bytes, `ControlPlane.registerDeviceKey` stores only `DeviceIdentity` metadata, `displayDeviceKeyFingerprint` returns the user-visible fingerprint/status surface, `getDeviceKeyStatus` projects the same identity to `GovernanceKeyStatus`, and `revokeDeviceKey` moves the identity to revoked without storing or exposing private key material. For private key custody, `DevicePrivateKeyStore.provisionDevicePrivateKey` creates or accepts an Ed25519 keypair, stores only the private PEM in an injected credential store behind `keychain://archcontext/device/<account>/<publicKeyId>`, and returns a public `DeviceKeyCredentialReference` plus public key. `signWithDevicePrivateKey` reads the credential-store ref, signs canonical payload bytes, and never accepts repository paths, file URLs, or config refs as private-key locations. For GitHub connect, the CLI path is `archctx github connect` into `runGithubCommand`: it creates an OAuth PKCE authorization URL, stores the PKCE verifier and refresh token behind keychain refs, provisions a Device private key credential ref, writes only refs/fingerprints/scopes to `.archcontext/.local/github-connection.json`, and reports `ghCli: "not-used"`. `archctx github status` reads that safe record without starting a daemon or invoking `gh`; `archctx github disconnect` removes the connection record and clears credential refs. For Challenge state changes, the path is current Challenge status plus requested target status, through `assertCanTransitionChallenge`, into a new immutable ReviewChallenge v2 value or a deterministic rejection. For new-head arrival, the control-plane path takes the next Challenge and existing ReviewChallenge v2 rows, selects only same installation/repository/PR, different head, and active statuses (`PENDING`, `LEASED`, `SUBMITTED`), then transitions those old values to `SUPERSEDED` through the same contracts guard. For lease claim, the control-plane path receives a Challenge, claimant ID, current lease metadata, and current time; it grants a five-minute metadata-only lease capped by Challenge expiry, blocks a different claimant while an active lease exists, allows same-owner renewal, allows expired-lease reclaim, and rejects terminal or expired Challenges. For legacy migration, a signed Attestation v1 can only become an `archcontext.attestation-migration/v1` audit record with `requiredCheckEligible=false` and `ATTESTATION_SCHEMA_UNSUPPORTED`; it is not converted into an Attestation v2. For Attestation submit, the control-plane path receives a leased Challenge, an unknown runtime payload, current time, and caller-provided consumed nonce hashes; it rejects Attestation v1 or invalid schema before required-check evaluation, rejects consumed nonce replay, expired Challenge/Attestation, superseded Challenge, nonce mismatch, identity mismatch, and trust mismatch without changing the nonce set, or returns a `SUBMITTED` Challenge plus a new consumed nonce hash set on success. For persistence, the D1 schema stores Device Key metadata, installation ID, repository ID, PR number, head SHA, required trust, status, time bounds, lease metadata, nonce hash, v2 Attestation metadata, and legacy v1 audit-only migration metadata, then enforces active Challenge identity uniqueness with a partial unique index.

For pull-head verification, the traced path is `archctx github verify-head --challenge-json <ReviewChallengeV2>` into `readReviewChallengeV2Arg`, which validates the JSON with `assertReviewChallengeV2` before any GitHub metadata fetch. `runGithubCommand` then calls `ControlPlane.fetchAndVerifyReviewChallengePullHead`, which fetches only `getPullHeadMetadata({ installationId, repositoryId, pullRequestNumber })`; the control-plane compares observed installation/repository/PR/head/base values against the Challenge and returns `accepted:true` or a deterministic `REPOSITORY_MISMATCH`, `PULL_REQUEST_MISMATCH`, `HEAD_SHA_MISMATCH`, or `BASE_SHA_MISMATCH`. The CLI output contains expected/observed metadata plus `challengeId` and `ghCli:"not-used"`; it does not include the Challenge nonce, source, diff, patch, file paths, findings, PKCE verifier, tokens, or private keys.

For installed-bin runtime verification, the traced path is installed `archctx sync` into the runtime daemon, then `CodeGraphCliProvider.run(["sync", root])`. The provider resolves `codegraph` from `PATH`; when it resolves to a JavaScript shim such as `node_modules/.bin/codegraph -> @colbymchenry/codegraph/npm-shim.js`, it invokes `process.execPath <resolved-shim> ...args` under the same CLI runtime. Native binaries and unresolved custom names still use the direct command path. The local product E2E and no-cloud script now initialize CodeGraph through the same resolved shim path before asserting `doctor → init → sync → context/review`.

For detached worktree preparation, the traced path is `ArchctxDaemon.prepareDeveloperReviewWorktree({ repositoryRoot, challenge, expectedHeadTreeOid })` into `prepareDetachedReviewWorktree`. The git adapter resolves the source repository root, verifies `challenge.headSha^{commit}` exists, computes or checks the expected `headSha^{tree}`, creates `git worktree add --detach <tmp>/worktree <challenge.headSha>`, then verifies `rev-parse HEAD`, `rev-parse HEAD^{tree}`, detached HEAD state, and tracked clean state with `git diff --quiet` and `git diff --cached --quiet`. Accepted output carries only the local temp worktree path plus head/tree/clean/detached metadata. Unavailable heads, wrong expected tree OID, non-detached roots, and dirty tracked worktrees fail with deterministic local reason codes before downstream Review/Attestation work can continue.

For digest calculation, the traced path is `ArchctxDaemon.computeDeveloperReviewDigestBundle({ challenge, worktree })`. The daemon first re-runs detached worktree verification against the Challenge head and recorded tree OID. Only a matching current head can proceed to Attestation v2 evaluation, expiry checks, nonce consumption, and `LEASED -> SUBMITTED`.

For server Attestation verification, the traced path is `ControlPlane.submitReviewChallengeAttestation({ challenge, attestation, currentPullHead, publicKey, now, expectedHeadTreeOid, consumedNonceHashes })`. The control-plane first verifies the current PR head against the Challenge, then calls `verifyAttestationV2ForReviewChallenge`. The verifier rejects invalid or private-content payloads, Attestation v1, Challenge ID mismatch, repository/PR/head/base mismatch, nonce mismatch, required-trust mismatch, expired Challenge or Attestation, optional tree OID mismatch, and invalid Ed25519 signatures over `canonicalAttestationV2`. Because the canonical payload blanks only `signature.value`, tampered worktree/model/policy/codeFacts/review digests and tampered tree values fail signature verification unless an expected tree mismatch rejects earlier. Only a signed, matching payload returns `attestationDigest`; nonce consumption and `LEASED -> SUBMITTED` happen after that accepted result.

For FG3-20 security rejection, the traced path is the same submit call with `signingKeyStatus` included. The control-plane still checks current PR head first, then the verifier checks Challenge identity, nonce, trust, expiry, expected tree, signing-key status, zero digest values, and signature before nonce consumption. A revoked Device Key returns `DEVICE_REVOKED`; a revoked Runner key returns `RUNNER_REVOKED`; a public-key ID mismatch between Attestation execution metadata and server key status returns `SIGNATURE_INVALID`; any all-zero Attestation or runtime digest returns `DIGEST_INVALID`; and wrong Developer-vs-Organization trust returns `TRUST_LEVEL_MISMATCH`. The control-plane security-matrix test proves repository, PR, head, base, tree, nonce, zero digest, wrong trust, revoked key, and replay rejections all leave the nonce hash unconsumed.

For Developer Review Check publication and FG3-EG6, the traced path is accepted Attestation v2 plus a Check Run ID into `GitHubAppState.updateDeveloperReviewCheckFromAttestation`. The helper loads the queued `ArchContext / Developer Review` Check, verifies the publication conditions (`accepted`, developer trust, same head, `result:"pass"`, and Developer Review check name), renders a GitHub Markdown summary, and completes the Check as `success` with output title `Developer-attested`. The E3 readback path is `bun run readback:fg3:developer-review`: it loads `_ops/env/fg2-staging.env` without printing secrets, creates a short-lived signed Attestation v2, verifies it through `verifyAttestationV2ForReviewChallenge`, fetches the real PR head through `GET /repositories/{repository_id}/pulls/{pull_number}`, creates and updates a real Check Run, reads that Check Run back, and writes only sanitized proof that the heading, `Developer-attested` title, `clean-commit-worktree` execution marker, Attestation digest marker, and zero forbidden source-code phrase matches were observed. A separate `gh pr checks 2` readback confirms the PR-visible context is the independent `ArchContext / Developer Review` Check and is currently passing.

For FG3-EG7 requiredTrust isolation, the traced path is the same real PR head plus two policy branches. The Developer branch creates a Developer-required Challenge and a signed Developer Attestation v2, verifies it successfully, and publishes the independent `ArchContext / Developer Review` Check as success. The Organization branch creates an Organization-required Challenge for the same PR head, signs a Developer Attestation v2, and verifies it through `verifyAttestationV2ForReviewChallenge`; the verifier rejects before publication eligibility with `TRUST_LEVEL_MISMATCH`, matching `satisfiesRequiredTrust("developer", "organization") === false`. The GitHub surface then publishes a separate `ArchContext / Organization Runner` Check as failure with `Attestation required`. The policy surface creates a temporary active repository ruleset targeting only `refs/heads/fg3-required-trust-smoke/*`, binds required status check context `ArchContext / Organization Runner` to App ID `4102781`, reads it back, deletes it, and confirms no same-prefix ruleset remains.

For FG3-EG8 DLP, the traced path is evidence and runtime projection rather than a new product endpoint. `scripts/fg3-cloud-dlp-readback.ts` reads the eight verified FG3 artifacts that cover process review, negative identity, adversarial conclusion, attestation security, supersede, Developer Check provenance, real PR synchronize, and requiredTrust isolation. For Check supersede publication, the traced path is `pull_request.synchronize` or a newer PR head into `GitHubAppState.handlePullRequest`, which calls `invalidateHead` before queuing the new Check. ## P3 Decision

The implementation reuses `@archcontext/contracts` canonical JSON and digest helpers instead of introducing a separate attestation canonicalizer. `ControlPlane.issueReviewChallenge` owns crypto nonce issuance and default Challenge expiry while still delegating value validation to `createReviewChallengeV2`. `createAttestationV2` constructs unsigned signed-shape payloads, and FG3-17 routes daemon signing through the Device private-key credential reference without exposing key material. Device Key registration stays public-key metadata only, and Device private-key custody is isolated behind a credential-store port so repo files, ordinary config, MCP output, SQLite, and D1 can carry only refs/fingerprints. The GitHub CLI connect state is intentionally a local safe record rather than a GitHub API integration: FG3-11 proves the product command and credential boundary without depending on `gh`, while FG3-12 owns Challenge fetch/head verification and FG5 owns durable cloud API exchange. Challenge transition, supersede, lease, and nonce consumption logic stays pure and side-effect free so durable storage and GitHub Check publication can reuse one transition guard without duplicating rules. Lease state is modeled separately from `ReviewChallengeV2` because the PRD stores `lease_owner` and `lease_expires_at` in `review_challenges`, not in the Challenge DTO. Attestation v1 migration is intentionally audit-only to preserve historical evidence without silently maintaining a dual-track verifier; new required checks must pass through Attestation v2 evaluation. The nonce consumer accepts a caller-provided hash set and returns the next hash set only on success; FG5 will bind that contract to a real D1 transaction with append-only Attestation persistence. The D1 unique index is scoped to active statuses (`PENDING`, `LEASED`, `SUBMITTED`) so terminal rows do not block legitimate retry policy while active duplicates for the same head/trust are impossible; the control-plane supersede rule closes the cross-head active Challenge case. This preserves the invariant that challenge, device, GitHub connection, and attestation payloads contain metadata and digests only, not source, diff, file paths, findings, review content, raw tokens, PKCE verifier, or private keys.

FG3-12 keeps the GitHub pull-head check as a metadata identity gate instead of a partial worktree executor. The worktree lifecycle has stricter responsibilities: fetching git objects, detached checkout, HEAD/tree/clean verification, CodeGraph sync, review execution, signing, and cleanup. Those begin at FG3-13. By stopping FG3-12 at `ReviewChallengePullHeadVerification`, the next slice receives a precise, nonce-free identity result while the current slice still proves that stale or wrong PR heads cannot pass into runtime execution.

The CodeGraph shim hardening is intentionally placed in the existing runtime adapter rather than the FG3 worktree runner. It fixes the current installed-product execution boundary discovered by full verification without broadening FG3-12 into detached worktree execution. At 10x, the first failure mode would be repeated process invocation overhead; that is acceptable for this slice because CodeGraph calls remain coarse-grained CLI operations and the adapter can cache an invocation resolution later if profiling shows pressure.

FG3-13 keeps worktree preparation separate from digest/review/signing. This preserves the invariant that a Developer Attestation can later claim `execution.source="clean-commit-worktree"` only after an isolated detached checkout has been proven exact and clean. The current source checkout may be dirty; that does not taint the accepted review worktree because execution is bound to the detached commit. FG3-23 now owns the temporary run lifecycle that preserves this clean-state gate during success, failure, and stale-run recovery.

FG3-14 keeps digest bundling separate from deterministic review. `codeFactsDigest` is derived from a CodeFacts snapshot boundary rather than review output, so FG3-15 can initialize/sync real CodeGraph inside the detached worktree and then reuse the same digest bundle shape. The worktree digest uses Git tree object IDs rather than filesystem walking because the PRD requires tracked tree binding and says untracked files must not affect publishability unless the review depends on them. At 10x scale, this makes digest cost proportional to tracked Git entries and avoids hashing generated or untracked local state.

FG3-15 keeps the review conclusion inside the daemon-owned Review Engine path rather than accepting a caller-provided result or digest. `runDeveloperReviewSession` returns `reviewDigest` only after `completeTaskGate` produces and digests the review object, and local persistence stores the detailed result for later signer and submit steps. At 10x scale, the first pressure point is repeated CodeFacts sync per temporary worktree; this is acceptable for the security boundary because the review session must bind to the exact Challenge checkout, and later caching can only be added behind the same worktree identity.

FG3-16 moves local completion away from caller-supplied digest inputs instead of merely rejecting `result`. This preserves the PRD §9.7 invariant that Attestation fields are daemon-owned: the caller can identify the task and expected head, but cannot provide pass/fail, review digest, policy/model digest, or signature material. The core Review Engine still accepts an internal `CompleteTaskInput` with daemon-computed digests so existing deterministic Review logic stays pure and testable. At 10x scale, repeated daemon sync on completion is the first cost pressure; that is acceptable here because correctness requires digest ownership before FG3-17 signing.

FG3-17 keeps signing in local-runtime behind a contracts-owned signer port instead of importing `@archcontext/cloud/attestation` into the daemon. That preserves the package boundary: contracts own stable canonical bytes, cloud owns server-side evaluation, local-runtime owns private-key use. The Device private key remains behind the credential-store signer implementation; daemon outputs contain public identity, digest, and signature metadata only. At 10x scale, the first pressure point is key lookup/sign latency per Attestation, which is acceptable before submit/retry orchestration because signing must remain after exact-head review and before server verification.

FG3-18 places the race guard in the control-plane submit function rather than only in the CLI preflight. That makes the invariant enforceable for future CLI, daemon, runner, and durable API callers: a submit attempt must carry or fetch current pull-head metadata before nonce consumption. The existing pure head-verification helper remains the comparison source, so FG3-12 and FG3-18 do not drift. At 10x scale, the first pressure point is an extra GitHub metadata call per submit; this is acceptable because stale-head acceptance is a correctness failure, and the call is cheaper than running the deterministic review again.

FG3-19 keeps cryptographic verification in `@archcontext/cloud/attestation` rather than in the control-plane. That preserves one reusable verifier for Developer Runtime submit and future Organization Runner submit, while control-plane owns submit ordering: current-head recheck first, server verifier second, nonce consumption third. At 10x scale, the first pressure point is public-key lookup and Ed25519 verification per submit; this is acceptable for an E2 verifier slice, and durable key lookup/revocation policy remains a later security-matrix task.

FG3-20 keeps key status as an explicit verifier input instead of making the attestation package query control-plane storage. That preserves package direction and lets FG5 wire the same contract to D1 transactions later. Zero-digest rejection is semantic, not schema-only: `sha256:000...` is syntactically valid but is not acceptable as evidence for a required check. At 10x scale, the first pressure point is avoiding repeated key-status reads per submit; that belongs with durable transaction/idempotency work, while this slice closes the correctness rule and the adversarial test matrix.

FG3-21 keeps GitHub Check publication as a GitHub App concern instead of embedding Check API calls into the verifier or control-plane submit function. That preserves the invariant that `@archcontext/cloud/attestation` verifies evidence, while `@archcontext/cloud/github-app` owns user-visible Check semantics. The staging readback intentionally records only sanitized booleans and request IDs, not the Markdown summary, tokens, JWTs, signatures, or private keys. At 10x scale, the first pressure point is idempotent Check delivery and retry/DLQ handling; that belongs to FG5, while this slice proves the user-visible Developer Review provenance and GitHub App permissions path.

FG3-22 uses `neutral` plus explicit supersede copy rather than introducing an internal `stale` conclusion enum. GitHub's Check API does not allow Apps to write `stale`, so the product state remains domain-level (`SUPERSEDED`) while the GitHub user-facing surface is an allowed neutral/cancelled-style conclusion. At 10x scale, the first pressure point is making the old-check update and new-check creation transactional and retryable; that belongs to FG5 durable Check Delivery, while this slice proves the public GitHub UX contract.

FG3-23 keeps lifecycle ownership in the runtime daemon instead of expanding the git adapter or CodeGraph adapter contracts. The git adapter still owns exact worktree creation/removal primitives, and CodeGraph remains a metadata/facts port; the daemon owns the multi-step run because it is the only boundary that sees Challenge identity, worktree creation, CodeGraph sync, review execution, signing, and the FG3-24 submit/retry UX. At 10x scale, the first pressure point is coordinating local run locks with cloud Challenge leases and durable Check delivery; that belongs to FG5, while this slice closes local leak/recovery semantics.

FG3-24 keeps the CLI orchestration honest about transport. It does not synthesize a SaaS Challenge or pretend that production submission has happened when no Challenge/submission port is configured. Instead, the command can run against current injected ports and leaves `ready_for_submit` when the production transport is absent. This preserves the invariant that local review/signing is real and daemon-owned, while cloud persistence/check delivery remains a later durable integration. At 10x scale, the first pressure point is idempotent server-side submit and Check delivery retry/DLQ; that belongs to FG5.

FG3-EG1 treats process evidence as a fixture boundary rather than a staging claim. It proves the local exact-head clean temporary worktree and deterministic Review path in a separate command process, while avoiding any claim that production Challenge retrieval or durable Check delivery is complete. The dirty source checkout in the fixture is intentional: it proves the Review result is bound to the detached commit, not to whatever edits are present in the user's working tree.

FG3-EG2 keeps identity rejection at the boundary that owns each invariant instead of adding a broad catch-all validator. Current PR repository/head/base identity rejects in CLI claim through `ControlPlane.fetchAndVerifyReviewChallengePullHead`; tree mismatch rejects in detached worktree preparation before review; dirty detached worktree rejects in daemon digest calculation before signing. This preserves the invariant that only a Challenge-bound, exact, clean commit worktree can produce Developer Review evidence, while a dirty source checkout remains allowed because it is not the execution worktree.

FG3-EG3 keeps pass/fail ownership in the deterministic Review path instead of trying to sanitize forged conclusions after the fact. The contracts denylist is recursive and surface-neutral; CLI and MCP reject before runtime RPC, and the daemon repeats the same guard before local completion or Developer Review Attestation signing. At 10x scale, the first pressure point is keeping new Agent surfaces aligned with this denylist; the current evidence makes that a contract-level invariant rather than a one-off CLI check.

FG3-EG4 keeps security rejection in the control-plane submit ordering instead of relying only on the lower-level verifier. Current-head verification and nonce-consumption checks run before server verification, while expiry, key status, nonce mismatch, and signature/digest semantics stay in `verifyAttestationV2ForReviewChallenge`. This preserves the invariant that only one live Challenge nonce can advance to `SUBMITTED`, and every rejected security case either observes the prior consumed set or leaves nonce state unchanged.

FG3-EG5 keeps live supersede at the GitHub metadata boundary rather than adding a generic GitHub client or code-content endpoint. The new `github.check-list-for-ref` operation is narrowly scoped to `/repositories/{repository_id}/commits/{ref}/check-runs?check_name=...`, records only egress category/path template/status metadata, and keeps files, contents, blob, tree, diff, and patch endpoints forbidden. This is sufficient to update user-visible stale Check state on synchronize while preserving the privacy invariant. It is not a substitute for FG5 durable Challenge/Check delivery transactions; the atomic Challenge storage cutover still belongs there.

FG3-EG6 keeps provenance in the Check summary rather than in PR comments, annotations, or status descriptions. The visible surface is a single independent GitHub Check named `ArchContext / Developer Review`, and the summary uses digest/provenance markers instead of source snippets or findings. At 10x scale, the first pressure point is stale or duplicate CheckRuns on the same head; EG5 supersede plus the latest Developer Review readback keeps the PR-visible context accurate, while durable de-duplication and retry ordering remain FG5.

FG3-EG7 keeps Organization trust as a higher requiredTrust policy rather than trying to infer it from a passing Developer Review. The user-visible guarantee is context isolation: `ArchContext / Developer Review` may be green, while `ArchContext / Organization Runner` stays failed until Organization Attestation exists. At 10x scale, the first pressure point is ruleset and CheckRun duplication during repeated staging probes; the evidence uses a tightly scoped temporary ruleset and deletes it immediately, while durable Organization Runner lifecycle remains FG4/FG5.

FG3-EG8 keeps DLP proof as a readback artifact rather than adding logging logic to the product path. The invariant is that Cloud-facing DTOs and logs carry metadata, digests, status, reason codes, Check IDs, and request categories, never source, diff, symbol, model body, or detailed finding content. At 10x scale, the first pressure point is keeping future durable queues and storage exports under the same scan; that broader storage-plane proof remains FG5/FG6, while FG3 now proves the Developer Review path and staging Worker logs.

- `assertReviewChallengeV2` rejects missing fields, unknown fields, invalid numeric IDs, invalid head/base OIDs, unsupported `requiredTrust`, unsupported status, invalid time ordering, and private-content keys.
