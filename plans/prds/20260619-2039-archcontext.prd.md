# 1. 文档控制与关键结论

## 1.1 文档目的

本文档用于冻结 ArchContext v2 的产品定义、系统边界、架构模型、运行时协议、隐私边界与 MVP 验收标准，并作为后续 Sprint、Agent 开发任务、Schema、接口契约和 ADR 的上游依据。

本文档替代此前将产品描述为“架构文档与索引工具”的定义。新的产品核心是：

> **ArchContext 是嵌入 Agentic Coding Runtime 的软件架构控制循环。**

架构文档、图表、ADR 和数据库均为该控制循环的输入或输出，而不是产品本身。

## 1.2 已冻结的关键决策

| 决策项 | 结论 |
|---|---|
| 核心定位 | Coding Agent 的主动架构运行时与软件工程 SOP 层 |
| 主要价值 | 让 Agent 知道何时局部修改、何时重构、如何抵达正确目标状态 |
| CodeGraph | 产品级硬依赖；代码层面通过 Adapter 软耦合；禁止读取其内部数据库 |
| 本地数据库 | SQLite；不采用 PGlite |
| 架构事实源 | Git 中 `.archcontext/` 的结构化模型、决策与策略 |
| 代码事实源 | CodeGraph + Git / Worktree |
| 派生状态 | 本地 ArchContext SQLite，可重建，不提交 Git |
| 本地进程 | `archctxd` 长生命周期 Runtime Daemon；CLI 和 MCP 为薄适配层 |
| MCP | 本地 `stdio` 为完整能力；本地 Streamable HTTP 用于 Secure MCP Tunnel |
| ChatGPT App | 支持；完整能力经本地 Runtime + Secure MCP Tunnel；SaaS 远程 MCP 仅元数据 |
| GitHub App | 管理 Installation、PR 事件、Review Challenge 与 Check Run；默认无 Contents 权限 |
| SaaS | Cloudflare Workers + D1 + Queue；只处理身份、计费、GitHub 元数据和 Attestation |
| 定价 | 公开仓库免费；个人 Pro 5 美元/月，覆盖该开发者可访问的全部私有仓库 |
| Skills | 第一方 Skill 只描述 SOP；核心业务逻辑全部在 Runtime Core |
| Slack | 不进入 MVP；仅保留 Notification Port |
| 技术栈 | TypeScript + Node.js 24 LTS + Bun workspaces；MCP TypeScript SDK v1.x |

## 1.3 产品不变量

以下规则不得由实现便利、Agent 建议或单个客户需求绕过：

1. ArchContext SaaS 不接收源代码、Diff、Symbol、CodeGraph、架构模型正文或详细 Review Finding。
2. Agent 不得直接自由修改架构模型；所有结构化写入必须经过 ChangeSet、Schema、Policy 和原子提交。
3. 任何新增兼容层必须绑定真实契约、负责人、移除条件和复审时间。
4. 目标状态与迁移状态必须分离；中间兼容状态不能默认为最终架构。
5. 架构结论必须能追溯到声明意图、代码证据或显式人类决策。
6. 每个任务结果必须绑定 Repository、HEAD SHA 和 Worktree Digest，避免使用过期上下文。
7. 高结构压力时，不允许 Runtime 仅以“最小 Diff”作为默认优化目标。
8. 没有足够证据时，不继续叠加补丁；进入 Proof Point / Investigation 流程。
9. 简单项目不得被迫采用复杂企业架构模型；治理强度随项目复杂度渐进增长。
10. 产品必须在没有自有托管模型推理的情况下工作；推理来自用户正在使用的 Coding Agent。

# 2. 产品摘要

## 2.1 一句话定义

**ArchContext 是 Coding Agent 的架构控制运行时：它利用 CodeGraph 感知代码，在任务前编译架构上下文，在开发中检测结构压力，在必要时推动重构，并在任务完成前同步和验证系统状态。**

## 2.2 用户价值

用户不需要具备完整的软件工程、架构治理或文档管理知识，也不需要手工维护复杂的 C4、ADR、依赖边界或迁移计划。

用户只需要表达产品目标，例如：

> “增加团队订阅、邀请和角色权限。”

ArchContext 与 Coding Agent 协作完成：

- 定位相关代码与历史决策；
- 识别产品能力、数据所有权与安全边界；
- 判断当前结构能否承载需求；
- 阻止无依据的 wrapper、fallback 和双轨实现；
- 必要时生成目标状态、Proof Point、迁移方案与 Kill List；
- 将架构变化写入结构化模型与 ADR；
- 清理迁移完成后的旧路径；
- 在 GitHub 上发布绑定当前 Commit 的 Review 证明。

## 2.3 用户可感知的最终效果

```text
普通 Coding Agent：
完成当前任务，优先避免大范围修改。

CodeGraph：
快速告诉 Agent 代码在哪里、如何调用、改变会影响什么。

ArchContext：
主动判断当前任务是否暴露了结构问题，给出正确目标状态，
并把软件工程约束嵌入 Agent 的整个执行过程。
```

## 2.4 核心产品对象

ArchContext 的一等产品对象不是“文档”，而是：

| 对象 | 作用 |
|---|---|
| Task Context | 当前任务所需的最小完整架构上下文 |
| Architecture Posture | Normal、Structural、Intervention 或 Proof Required |
| Architecture Intervention | 对必要结构性改变的论点、目标、约束、验证与清理计划 |
| Compatibility Contract | 对临时或长期兼容代码的合法依据与移除机制 |
| Architecture ChangeSet | 可验证、可预览、可回滚的架构文件变更 |
| Review Result | 对实现、架构意图、目标状态与技术债的综合验证 |
| Attestation | 不上传代码情况下，绑定 Commit SHA 的签名 Review 证明 |

# 3. 背景、问题与机会

## 3.1 Vibe Coding 的增长瓶颈

Vibe Coding 在原型阶段表现极佳：用户描述需求，Agent 快速生成可运行代码。但当产品持续增长时，开发失败通常不是因为 Agent 不会生成代码，而是因为系统缺少长期的软件工程控制层。

典型演进路径：

```text
第 1 周：一个页面、几个 API，修改速度很快
第 1 月：认证、数据库、支付、后台任务开始耦合
第 3 月：同一概念出现多个实现，新旧接口并存
第 6 月：每个需求都需要绕过旧代码，Agent 不敢删除任何路径
最终：项目在失败、局部修补和大规模重写之间循环
```

## 3.2 Agent 的局部最优倾向

Agent 在缺少架构上下文时，会理性地选择局部风险最低的策略：

- 保留旧 API，再添加 V2；
- 新增 wrapper 而不更新调用方；
- 同时支持新旧字段；
- 引入 fallback，避免删除旧路径；
- 复制逻辑以减少当前 Diff；
- 将迁移中间态永久保留；
- 把“兼容性”作为无需证据的默认理由。

这些策略在单次任务中看似谨慎，但长期造成：

- 责任边界模糊；
- 生命周期拥有者重复；
- 数据所有权失控；
- 调用链、条件分支和迁移路径不断增长；
- Agent 每次需要理解更多历史偶然性；
- 重构风险和成本持续上升。

## 3.3 用户缺少的不是文档工具，而是软件工程判断

多数个人开发者和 Vibe Coding 用户无法可靠回答：

- 什么时候应该拆模块？
- 哪些约束是真实外部契约，哪些只是历史惯性？
- 哪些兼容代码必须保留，哪些应该立即删除？
- 当前需求是实现问题，还是暴露了架构问题？
- 数据应该由谁拥有？
- 什么时候需要迁移而不是补丁？
- 如何让下一次 Agent Session 理解今天的决定？

要求这些用户手工维护架构文档，本质上是把架构师的工作交给了最缺少相关经验的人。

## 3.4 CodeGraph 解决了“看见代码”，但不会自动形成工程 SOP

CodeGraph 提供本地代码知识图谱、符号、调用链、影响范围和任务上下文，是 ArchContext 的关键代码事实依赖。它能回答：

- 某个函数在哪里；
- 谁调用它；
- 改动会影响哪些 Symbol；
- 某条执行链如何经过多个模块。

ArchContext 负责进一步回答：

- 这条调用是否符合系统边界；
- 当前责任应该归哪个模块；
- 哪些约束是真实契约；
- 是否应该进行结构性重构；
- 应该删除哪些旧概念；
- 如何验证迁移已真正完成。

## 3.5 市场机会

ArchContext 的机会不是成为另一个画图、架构文档或通用 Code Review 产品，而是成为：

> **Agentic Coding 时代的软件工程控制层。**

其价值随着项目生命周期增长而上升：项目越复杂、Agent Session 越多、历史决策越长，ArchContext 的复利价值越明显。

# 4. 产品愿景、定位与原则

## 4.1 愿景

让任何开发者都能从一个简单 Web 应用出发，依靠优秀 Coding Agent 构建并长期维护复杂产品，而不必先成为软件架构师。

## 4.2 市场定位

### 主标语

> **Code with an architect on standby.** —— 你的 Coding Agent 背后随时有一位资深架构师待命：在该重构时给出有依据的正确改动，而不只是最安全的小补丁。

### 面向用户的定位

> **从第一个页面到复杂产品，让你的 Agent 始终知道系统应该怎样工作。**

### 面向开发者生态的定位

> **ArchContext is the architecture runtime for coding agents.**

### 与 CodeGraph 的关系

> **CodeGraph 是 Agent 的代码感知层；ArchContext 是 Agent 开发 SOP 中的架构判断与控制层。**

## 4.3 产品原则

### P-01：Agent-first，而非 diagram-first

所有核心 Schema、工具和流程优先服务 Agent 的检索、决策、变更与验证。图表和 Markdown 是面向人的投影。

### P-02：正确目标优先于最小 Diff

Runtime 不能将“少改代码”默认为最优方案。应综合长期复杂度、真实约束、迁移成本和可验证收益。

### P-03：有证据的架构勇气

既反对无证据的激进重写，也反对无证据的兼容保留。结构性改变必须具备 CodeGraph Evidence、明确目标和 Proof Point。

### P-04：Progressive Architecture

只在复杂度出现时增加治理。简单项目以最小模型开始，复杂系统再逐步引入边界、契约、迁移和 Review Gate。

### P-05：Repository 是协作事实源

结构化架构意图、ADR、Policy 和 Compatibility Contract 存于 Git；本地数据库仅是派生索引。

### P-06：Declared / Observed / Verified 分离

用户声明的目标架构、代码实际表现和双方验证状态不得混为一体。

### P-07：兼容代码必须持证上岗

每个兼容层必须说明保护对象、消费者、负责人、移除条件和复审日期。

### P-08：Local-first 与最小云端

代码智能、架构模型、详细 Review 和文件写入均在本地。SaaS 只承担身份、计费、GitHub 元数据与证明验证。

### P-09：薄工具面

对 Agent 暴露少量高层工具，一次返回任务所需完整上下文，避免要求 Agent 在大量原子工具间自行编排。

### P-10：确定性内核，Agent 负责判断表达

Schema、状态机、校验、路径安全、Digest、Policy 和写入必须确定性执行；自然语言分析和方案表达由用户当前 Agent 完成。

# 5. 目标用户与核心场景

## 5.1 Persona A：Vibe Coding 个人开发者

**特征：** 能表达产品需求，具备基础技术理解，但缺少系统架构与长期工程治理经验。  
**目标：** 从简单应用持续增长为真实产品。  
**主要痛点：** Agent 每次完成局部需求，但系统逐渐失控。

## 5.2 Persona B：高级个人开发者 / 开源维护者

**特征：** 理解软件工程，但同时使用多个 Agent，希望减少重复解释和 Review 成本。  
**目标：** 让不同 Agent 遵循同一架构意图。  
**主要痛点：** Agent 不读取或无法正确综合 ADR、边界与历史决策。

## 5.3 Persona C：小团队中的独立贡献者

**特征：** 使用团队私有 Repository，有 GitHub PR 流程。  
**目标：** 在代码不进入第三方分析平台的前提下获得架构 Review。  
**主要痛点：** 团队 Review 依赖少数资深工程师，架构问题发现过晚。

## 5.4 主要 Jobs to Be Done

1. 当我开始一个任务时，让 Agent 自动得到相关架构、约束和历史决策。
2. 当局部补丁会继续积累技术债时，主动提醒并生成可证明的重构路径。
3. 当 Agent 引入兼容代码时，要求它说明真实契约和移除条件。
4. 当代码结构变化时，自动同步架构模型，而不是让我维护文档。
5. 当任务完成时，验证新实现是否符合目标状态，并清理旧路径。
6. 当提交 PR 时，在不上传源代码的情况下提供架构 Review Check。
7. 当我在 ChatGPT 中讨论产品或架构时，允许 ChatGPT 通过安全本地连接读取任务级架构上下文。

# 6. 产品目标、非目标与范围

## 6.1 MVP 目标

- 支持单个 Git Repository 的本地架构控制循环。
- 将 CodeGraph 作为必需代码事实引擎。
- 支持从简单项目自动建立最小架构模型。
- 支持 `prepare -> checkpoint -> intervention/change -> complete` SOP。
- 支持 Normal、Structural、Intervention、Proof Required 四类建议结果。
- 支持 Architecture Pressure 与 Refactor Confidence 的可解释计算。
- 支持 Compatibility Contract、Architecture Intervention 和 ChangeSet。
- 支持安全原子写入 `.archcontext/`。
- 支持本地 stdio MCP、CLI 和 Agent 指令安装。
- 支持 ChatGPT Secure MCP Tunnel 连接本地 Runtime。
- 支持 GitHub App Challenge、签名 Attestation 和 Check Run。
- 支持公开仓库免费、个人私有仓库统一 5 美元/月。

## 6.2 MVP 非目标

- 不开发通用代码图谱解析器；使用 CodeGraph。
- 不开发大型可视化架构画布。
- 不提供云端代码分析、云端 Embedding 或托管 Repository Clone。
- 不提供通用 Bug / Security AI Code Review。
- 不支持跨 Repository 架构图谱。
- 不支持组织级强制可信 Runner。
- 不支持 Slack Bot。
- 不支持多人实时协作编辑器。
- 不引入 PGlite、Vectorize 或托管 Vector DB。
- 不让 Agent 自动合并 PR。
- 不将第三方 Skill 作为核心执行依赖。

## 6.3 v1 后续范围

- Multi-repo Architecture Context。
- Organization Runner Attestation。
- Team / Enterprise Governance。
- LikeC4 / Structurizr 输出 Adapter。
- Slack、Webhook 和 Email 通知 Provider。
- 浏览器 Architecture Explorer。
- Trusted Execution / Device Integrity 增强。
- GPT App 发布目录与更完整交互 UI。

# 7. 渐进式架构体验

## 7.1 治理级别

ArchContext 自动根据代码规模、依赖复杂度、数据敏感度、外部接口和历史变化提升治理级别。用户可以覆盖自动判断，但不应被要求手工选择所有架构概念。

| Level | 典型项目 | 最小维护内容 | 主要控制 |
|---|---|---|---|
| L0 Prototype | 单页、小型 API、一个 DB | 产品目的、主要模块、外部服务 | 上下文与基础责任识别 |
| L1 Application | 认证、支付、后台任务、多个功能 | 能力、模块、数据所有权、外部接口 | 依赖与数据边界 |
| L2 Product | 多业务域、异步事件、复杂权限 | 契约、约束、ADR、迁移与兼容合同 | Intervention、Drift、架构 Review |
| L3 System | 多 Repository、团队与独立部署 | 跨仓关系、Owner、组织 Policy | Trusted Runner 与组织治理；非 MVP |

## 7.2 最小启动体验

`archctx init` 不要求用户填写架构模板，而是：

1. 初始化 CodeGraph。
2. 扫描框架、入口、数据存储和外部服务。
3. 推导初始模块和产品能力。
4. 询问最多 3-5 个非技术问题。
5. 生成最小 `.archcontext/` 模型。
6. 安装对应 Agent 的 MCP、Skill 或规则。

示例问题：

- 这个产品主要帮助谁完成什么？
- 哪些功能处理真实付款、身份或敏感数据？
- 哪些目录是生成代码或第三方代码？
- 当前是否已有对外承诺的 API 或数据格式？

## 7.3 面向非专家的 Human Gate

禁止直接向普通用户询问底层 Edge 或 YAML 字段。应将架构问题翻译为产品后果。

错误示例：

> 是否批准 `service.order -> datastore.payment` 的 `reads` 关系？

正确示例：

> 订单功能现在会直接读取付款数据。这会让订单与付款紧密绑定，并增加未来迁移风险。推荐由付款模块提供状态查询接口。请选择：
>
> 1. 使用推荐边界；
> 2. 暂时保留，并设置移除条件；
> 3. 查看技术细节。

## 7.4 自动复杂度升级条件

以下任一信号可触发治理升级建议：

- 新增第二个数据存储；
- 出现身份、支付或敏感数据；
- 新增外部公开 API；
- 出现异步队列或事件；
- 模块跨边界直接读写数据；
- 同一业务概念出现多个 Owner；
- 连续任务重复修改同一热点区域；
- 引入长期兼容路径；
- 一个简单需求的影响范围跨越多个不相关模块。

# 8. Agentic Coding 架构控制循环

![Agentic Coding 架构控制循环](archcontext_prd_v2_assets/sop_loop.png)

## 8.1 控制循环阶段

### Stage 1：Prepare Task

输入：用户任务、Repository、HEAD、Worktree。  
输出：Task Context、修改姿态、相关节点、约束、历史决策、风险和必需检查点。

必须完成：

- CodeGraph 增量同步；
- 任务相关 Symbol 与调用关系检索；
- 架构节点、ADR、Policy、Compatibility Contract 检索；
- 当前结构压力评估；
- 当前证据覆盖与重构信心评估。

### Stage 2：Agent Implement

Agent 进行编码。ArchContext 不代替 Agent 写业务代码，但通过 MCP Server Instructions、Skill 和 Hook 约束执行顺序。

### Stage 3：Checkpoint

在一次有意义的修改后调用，重新同步 CodeGraph 并比较：

- 新增依赖；
- 责任变化；
- 数据所有权变化；
- 新旧路径并存；
- Compatibility Debt；
- 目标状态偏差；
- 受影响的架构模型与 ADR。

### Stage 4：Intervention / Change

当局部修改不再合理时，ArchContext 生成 Architecture Intervention 或 ChangeSet。

### Stage 5：Reconcile

将 Declared Architecture 与 Observed Code Facts 对齐，得到 Verified / Drift / Unknown 状态。

### Stage 6：Cleanup Gate

任务不能仅以“新路径可用”结束。必须验证：

- 旧路径是否删除；
- 临时双写是否关闭；
- shim 是否仍有合法合同；
- Kill List 是否完成；
- 迁移完成条件是否满足；
- 未完成项是否被明确记录和限期。

### Stage 7：Complete Task

生成 Review Result、架构摘要、用户可理解的变更说明和可选 Attestation。

## 8.2 修改姿态

| 姿态 | 适用条件 | Runtime 行为 |
|---|---|---|
| Normal | 结构压力低，责任和边界不变化 | 提供局部上下文和常规完成检查 |
| Structural | 存在有限内部结构问题，影响范围可控 | 允许移动、合并、删除内部概念；要求更新调用方 |
| Intervention | 结构压力高且证据充分 | 必须先定义目标状态、迁移、Proof Point 和 Kill List |
| Proof Required | 结构压力高但证据不足 | 禁止继续堆补丁；先完成最小验证实验或调用方盘点 |

## 8.3 Architecture Pressure

Architecture Pressure 是可解释信号集合，不是单一黑盒 LLM 分数。内部可归一化为 0-100，但必须返回组成信号。

首版信号：

- Duplicate Responsibility；
- Multiple Lifecycle Owners；
- Unjustified Compatibility Path；
- Parallel New / Legacy Flow；
- Direct Cross-boundary Data Access；
- Circular Dependency；
- Repeated Hotspot Change；
- Excessive Change Radius；
- Temporary Migration State Overdue；
- Public Contract Without Owner；
- Architecture Drift Repeatedly Ignored。

## 8.4 Refactor Confidence

重构信心来自可验证覆盖：

- CodeGraph 调用方覆盖；
- 外部接口与持久化数据盘点；
- 测试覆盖；
- 回滚点；
- 目标模型清晰度；
- Proof Point 可执行性；
- 未知消费者数量；
- 数据迁移可逆性。

## 8.5 决策矩阵

| Architecture Pressure | Refactor Confidence | 默认决策 |
|---|---|---|
| 低 | 任意 | Normal；不为重构而重构 |
| 中 | 低 | Structural 或增加证据 |
| 中 | 高 | Structural；允许自动清理内部债务 |
| 高 | 低 | Proof Required；禁止新增无依据兼容层 |
| 高 | 高 | Intervention；优先正确目标状态 |

## 8.6 Architecture Reframe Policy

ArchContext 内建一套不依赖外部 Skill 的高位判断协议：

1. 定义正确的目标状态，而不是从最小 Diff 出发。
2. 区分真实约束与历史惯性。
3. 明确 Thesis：系统应该改变成什么。
4. 生成 Kill List：哪些概念、路径和 shim 应被删除。
5. 生成 First Proof Point：最小可证实方向正确的切片。
6. 生成 Falsifiers：哪些证据出现时应推翻当前方案。
7. 生成 Benefit Ledger：复杂度、调用链、概念数和维护成本减少多少。
8. 将迁移计划与最终目标分开表达。

# 10. 总体系统架构

![ArchContext 总体系统架构](archcontext_prd_v2_assets/system_architecture.png)

## 10.1 架构风格

ArchContext 采用以下组合：

- **Local-first Data Plane**：所有代码与架构内容处理在用户本地。
- **Hexagonal Architecture**：Core 通过 Ports 使用 CodeGraph、Git、SQLite、Cloud API。
- **Long-lived Runtime Daemon**：避免每个 MCP Tool Call 重复初始化索引和数据库。
- **Thin Client Adapters**：CLI、stdio MCP、Local HTTP MCP 和 Agent Hook 共享同一个 Application Core。
- **Event-backed Control Plane**：GitHub 和 Stripe Webhook 通过 Cloudflare Worker + Queue 异步处理。
- **Attestation instead of Upload**：SaaS 验证签名证明，而不是重新执行代码分析。

## 10.2 组件与职责

| 层 | 组件 | 主要职责 | 明确不负责 |
|---|---|---|---|
| Agent Surface | Coding Agent / IDE | 需求理解、代码实现、自然语言方案 | 架构状态持久化与安全写入 |
| Local Adapter | CLI / stdio MCP / local HTTP MCP | 协议转换、输入校验、结果预算 | 业务规则 |
| Local Runtime | `archctxd` | Workspace 生命周期、SOP 状态、并发控制 | 云端 Repository Clone |
| Core | Context/Pressure/Decision/Change/Review | 架构判断与控制循环 | 通用代码解析 |
| Code Facts | CodeGraph | Symbol、Call Graph、Impact、Code Context | 架构意图与策略 |
| Repo State | `.archcontext/` | 声明架构、决策、约束、兼容合同 | 高频代码证据 |
| Local Derived State | SQLite | Session、Evidence、Verification、Review、FTS | 团队协作事实源 |
| GitHub Governance | GitHub App | PR Challenge、Check Run、Installation | 读取代码、写架构文件 |
| SaaS | Cloudflare Worker + D1 + Queue | OAuth、订阅、GitHub Metadata、Attestation | 代码与详细架构分析 |
| Optional GPT Surface | ChatGPT App + Secure MCP Tunnel | 架构讨论与交互式 UI | 绕过本地权限或隐私策略 |

## 10.3 核心数据流

### 本地日常开发

```text

  -> local MCP
  -> archctxd
  -> CodeGraph sync/query
  -> Architecture Model + SQLite
  -> Context / Posture / Checkpoint / Review
  -> Agent continues
```

### GitHub Review

```text
GitHub PR event
  -> SaaS Review Challenge
  -> local runtime fetches challenge
```

# 11. 本地 Runtime 详细架构

![本地 Runtime 组件架构](archcontext_prd_v2_assets/local_components.png)

## 11.1 为什么需要 `archctxd`

仅用“薄 MCP 每次 spawn CLI”会导致：

- 重复启动 Node 和 CodeGraph；
- 重复打开 SQLite；
- 多 Agent 同时使用时产生锁与重复 watcher；
- 无法持续维护 Task Session；
- 无法稳定支持 ChatGPT Secure MCP Tunnel；
- 每次 Tool Call 都重新加载模型和策略。

因此目标架构使用一个按用户运行的本地 Daemon：

- 按需启动，空闲后可退出；
- 同一用户只运行一个实例；
- 维护多个 Workspace Session；
- 通过 Unix Domain Socket / Windows Named Pipe 提供本地 RPC；
- 需要 GPT Tunnel 时，额外开启仅绑定 `127.0.0.1` 的 Streamable HTTP MCP；
- 所有客户端共享相同 CodeGraph Handle、SQLite Connection 和状态机。

## 11.2 进程模型

```text
Agent Host
  -> archctx mcp (stdio adapter)
        -> local RPC
            -> archctxd

CLI
  -> local RPC when daemon available
  -> embedded fallback for recovery-only commands

OpenAI tunnel-client
  -> 127.0.0.1:<ephemeral>/mcp
      -> archctxd local HTTP adapter
```

本地 HTTP MCP 不得：

- 绑定 `0.0.0.0`；
- 使用长期静态 Token；
- 暴露未经过 Tool Policy 的内部方法；
- 在关闭 Tunnel 后继续监听；
- 输出 Secret 或完整环境变量。

## 11.3 Workspace Session

每个 Workspace Session 由以下键确定：

```text
repositoryFingerprint
worktreePath
branchOrDetachedHead
```

Session 状态：

```text
CLOSED
OPENING
READY
SYNCING
STALE
APPLYING_CHANGESET
REVIEWING
ERROR_RECOVERABLE
ERROR_FATAL
```

并发规则：

- 多个 Reader 可同时查询 Context；
- 同一 Repository 同时只允许一个 ChangeSet Apply；
- CodeGraph Sync 与 Review 可排队；
- Apply 期间拒绝基于旧 Worktree Digest 的请求；
- 每个 Tool Response 返回 Session Version。

## 11.4 Local Gateway

职责：

- 识别调用通道：CLI、stdio MCP、local tunnel；
- 绑定调用者权限；
- 限制 Result Budget；
- 过滤路径、Secret 与 Source 内容；
- 生成本地 Audit Event；
- 将通道策略注入 Use Case。

示例通道策略：

| 通道 | 读取源码 | 写模型 | 提交 Attestation | 默认结果预算 |
|---|---:|---:|---:|---:|
| CLI | 用户显式命令 | 是 | 是 | 32 KB |
| Local stdio MCP | 按工具策略 | 是，需 ChangeSet | 是 | 12 KB |
| ChatGPT Tunnel | 默认否；按请求摘要 | 默认仅计划 | 可选 | 8 KB |
| Cloud Metadata MCP | 否 | 否 | 否 | 4 KB |

## 11.5 SOP Orchestrator

SOP Orchestrator 是产品的主动层，负责把多个工具调用绑定为一个任务状态机。

Task Session 状态：

```text
CREATED
PREPARED
IMPLEMENTING
CHECKPOINT_REQUIRED
PROOF_REQUIRED
INTERVENTION_PROPOSED
CHANGESET_PLANNED
CHANGESET_APPLIED
RECONCILING
REVIEW_FAILED
READY_TO_COMPLETE
COMPLETED
ATTESTED
ABANDONED
```

状态转换约束：

- 未 `PREPARED` 的任务不得直接 `COMPLETE`；
- 高压力任务未解决 Intervention 时不得通过；
- ChangeSet Apply 后必须 Reconcile；
- Worktree 变化会使旧 Plan 标记 Stale；
- Cleanup Criteria 未满足时最多进入 `REVIEW_FAILED` 或显式 Deferred；
- Deferred 项必须有 Owner、Issue 和 Review Date。

## 11.6 Context Compiler

Context Compiler 将多源信息编译为任务级结构：

```text
Task Intent
+ CodeGraph Task Context
+ Changed Symbols
+ Declared Nodes / Relations
+ Constraints / ADR
+ Compatibility Contracts
+ Prior Interventions
+ Current Pressure Signals
= TaskContext
```

排序优先级：

1. Hard Constraint；
2. 当前任务直接相关模块；
3. 公共接口、数据和安全边界；
4. 活跃 Intervention / Migration；
5. 历史 ADR；
6. 代码 Evidence；
7. 辅助背景。

结果压缩规则：

- 默认不返回整份文档；
- 优先返回稳定 ID、摘要、理由和 Resource URI；
- Source 只返回最小必要片段；
- 重复关系合并；
- 超预算时保留约束和风险，优先移除背景说明。

## 11.7 Architecture Resolver

Resolver 维护三类状态：

```text
Declared：Repository 中声明的责任、关系、约束与目标
Observed：CodeGraph 与 Git 中观察到的实现事实
Verified：Declared 与 Observed 经验证后的匹配状态
```

验证状态：

| 状态 | 含义 |
|---|---|
| MATCHED | 声明与观察一致 |
| DRIFT | 代码与声明冲突 |
| DECLARED_ONLY | 有意图但缺少实现 Evidence |
| OBSERVED_ONLY | 代码关系存在但未声明 |
| UNKNOWN | CodeGraph 覆盖不足或无法判断 |
| OBSOLETE | 声明指向已不存在的实现 |
| MIGRATING | 当前处于明确迁移计划中 |

## 11.8 Pressure Analyzer

Pressure Analyzer 必须是“规则 + Evidence”的确定性组件，允许 Agent 解释，但不允许 Agent伪造输入。

每个 Signal 包含：

```json
{
  "type": "unjustified-compatibility-path",
  "severity": "high",
  "weight": 18,
  "evidence": ["symbol:LegacyStatusMapper", "path:src/..."],
  "explanation": "No compatibility contract or external consumer was found.",
  "confidence": 0.94
}
```

分数只用于排序，最终建议必须给出原始信号。

## 11.9 Refactor Decision Engine

输入：Pressure Signals、Evidence Coverage、Declared Intent、Constraints、Risk Profile。  
输出：Posture、Recommendation、Required Proof、Human Gate。

决策 Engine 不直接生成自然语言长文，而是生成结构化决策骨架，交由 Agent 组织表达。

## 11.10 Constraint & Policy Engine

首版 Policy 类型：

- dependency boundary；
- data ownership；
- public interface change；
- authentication/authorization boundary；
- compatibility contract；
- migration completion；
- architecture write path；
- required ADR；
- forbidden dependency；
- generated file protection。

Policy 规则可由 Repository 配置，核心安全 Policy 不允许关闭。

## 11.11 ChangeSet Engine

ChangeSet 只支持结构化 Operations：

```text
create_entity
update_entity_fields
delete_entity
move_entity_file
create_adr
update_adr_frontmatter
create_compatibility_contract
close_compatibility_contract
create_intervention
update_intervention_status
render_projection
```

禁止将“任意 shell command”或“任意文本 patch”作为核心 Operation。

Apply 流程：

1. 验证 Session、HEAD、Worktree Digest；
2. 验证所有 expectedHash；
3. 验证路径 allowlist 与 symlink；
4. 将 Operations 应用于内存模型；
5. 执行 JSON Schema；
6. 执行 Cross-entity Validation；
7. 执行 Policy；
8. 写入临时目录；
9. 生成 Diff；
10. 用户/Agent 确认；
11. atomic rename；
12. 更新 SQLite；
13. 重新 Render 与 Reconcile；
14. 失败则恢复原文件。

## 11.12 Reconciler & Renderer

Renderer 输出：

- `ARCHITECTURE.md`；
- `TREE.md`；
- `DEPENDENCIES.md`；
- `MIGRATIONS.md`；
- Mermaid 图；
- Agent Compact Context Snapshot。

Generated 文件由 Model 唯一生成，不允许手工修改。需要人工说明的内容放在 ADR 或节点的 `notes` 字段。

## 11.13 Review Engine

Review 维度：

1. Schema Validity；
2. Declared/Observed Drift；
3. Constraint Violation；
4. Architecture Pressure Regression；
5. Compatibility Debt；
6. Intervention Completion；
7. Target State Compliance；
8. Generated Projection Freshness；
9. Evidence Coverage；
10. Attestation Eligibility。

结论：

```text
PASS
PASS_WITH_NOTICES
FAIL_ACTION_REQUIRED
FAIL_STALE_CONTEXT
FAIL_INSUFFICIENT_EVIDENCE
```

## 11.14 Attestation Signer

每台设备生成 Ed25519 Key Pair：

- Private Key 保存在 OS Keychain 或系统凭据库；
- Public Key 注册至 SaaS；
- 每次 Attestation 使用 Challenge Nonce；
- Signature 覆盖 Repository ID、PR、HEAD SHA、Model Digest、Policy Digest、Result、Runtime Version、Timestamp 和 Nonce；
- SaaS 只保存必要证明，不保存 Review Detail。

# 12. CodeGraph 依赖与集成契约

## 12.1 依赖地位

MVP 明确规定：

> **ArchContext requires CodeGraph.**

CodeGraph 不是用户需要单独编排的第二套 MCP，而是 ArchContext 内部的代码事实引擎。

## 12.2 集成原则

```text
产品依赖：硬依赖
npm 版本：精确锁定
调用边界：CodeGraphAdapter
内部数据库：禁止访问
MCP 套 MCP：禁止
用户工具面：不暴露原始 CodeGraph Tool
升级方式：Contract Test + Golden Fixture
```

## 12.3 首选集成方式

优先使用 CodeGraph 公开 Node API：

```typescript
import CodeGraph from "@colbymchenry/codegraph";

const cg = await CodeGraph.open(workspaceRoot);
await cg.indexAll();
const context = await cg.buildContext(task, options);
const impact = cg.getImpactRadius(symbolId, depth);
```

CLI JSON Adapter 可作为隔离回退，但不能成为默认 MCP-to-MCP 设计。

## 12.4 ArchContext 最小依赖接口

```typescript
export interface CodeFactsPort {
  ensureReady(workspace: WorkspaceRef): Promise<CodeFactsSnapshot>;

  sync(input: {
    workspace: WorkspaceRef;
    changedPaths?: string[];
  }): Promise<CodeFactsSnapshot>;

  buildTaskContext(input: {
    task: string;
    maxSymbols: number;
    includeSource: boolean;
  }): Promise<NormalizedCodeContext>;

  findSymbols(query: SymbolQuery): Promise<NormalizedSymbol[]>;

  getImpact(input: ImpactQuery): Promise<NormalizedImpact>;

  getCallers(symbolId: string): Promise<NormalizedEdge[]>;

  getCallees(symbolId: string): Promise<NormalizedEdge[]>;

  resolveEvidence(selectors: SourceSelector[]): Promise<ObservedEvidence[]>;
}
```

## 12.5 归一化与防腐层

CodeGraph 的 Node ID、语言细节和结果格式不得进入 Architecture Domain。Adapter 负责转换为：

```text
NormalizedSymbol
NormalizedEdge
NormalizedCodeContext
NormalizedImpact
ObservedEvidence
CodeFactsSnapshot
```

## 12.6 版本策略

- `package.json` 使用精确版本，不使用 `^`；
- 每次升级运行跨语言 Contract Test；
- 维护最小兼容版本矩阵；
- 记录 CodeGraph Schema/Behavior Digest；
- 失败时阻止发布，而不是静默降级为不完整结果；
- 用户不得在未验证版本下提交 GitHub Attestation。

## 12.7 CodeGraph 生命周期

- 每个 Workspace 一个 Handle；
- Daemon 启动后惰性打开；
- 文件变化通过 watcher 增量更新；
- 多 Agent 共享同一 Handle；
- 空闲 Workspace 使用 LRU 关闭；
- `archctx doctor` 验证 Index Health、WAL、版本和同步状态。

# 13. CLI、MCP 与 Agent Runtime 集成

## 13.1 设计原则

CLI 是稳定、可测试、可脚本化的产品接口；MCP 是面向 Agent 的薄协议适配层。两者调用同一 Application Service，而不是 MCP 每次 spawn CLI。

```text
archctx CLI --------┐
                    ├-> Application Services -> Core

local HTTP MCP -----┘
```

```bash
archctx github review --pr 42
```

## 13.4 本地 MCP Tool Surface

默认只暴露五个高层工具：

### `archcontext_prepare_task`

- Read-only。
- 输入：task、workspace、budget、includeSource。
- 输出：TaskContext、Posture、Pressure、Confidence、Required Checkpoints。

### `archcontext_checkpoint`

- 读取当前修改并更新 Task Session。
- 输出：Impact、Pressure Delta、Required Intervention、Model Updates。

### `archcontext_plan_update`

- 生成 ChangeSet Draft；不写文件。
- 可根据 Agent Proposal、Intervention ID 或当前 Drift 生成。

### `archcontext_apply_update`

- 写操作；必须引用有效 ChangeSet。
- 默认需要 Human Confirmation。
- 具有 expectedHash、idempotency key 和 rollback。

### `archcontext_complete_task`

- 执行 Reconcile、Cleanup Gate 和 Review。
- 输出是否可以结束任务，以及剩余阻塞项。

```text

archcontext://constraint/{constraintId}
archcontext://intervention/{interventionId}
archcontext://compatibility/{contractId}
archcontext://changeset/{changeSetId}
archcontext://review/latest
archcontext://evidence/{evidenceId}
```

## 13.6 MCP Server Instructions

服务器初始化时向 Agent 提供跨工具规则：

```text
1. 在修改代码前调用 prepare_task。
2. 不要绕过 ArchContext 直接编辑 .archcontext 模型文件。
3. 有意义的代码修改后调用 checkpoint。
4. 不要新增 compatibility shim，除非存在 Compatibility Contract。
5. 当 Posture 为 Intervention 或 Proof Required 时，不要继续局部补丁。
6. 在结束任务或向用户宣称完成前调用 complete_task。
7. 遇到 stale context 时重新 prepare，而不是复用旧结果。
```

## 13.7 主动性的实现

MCP 本质上由 Client 调用，Runtime 无法假设所有 Agent 都会主动遵循 SOP。因此采用多层机制：

1. MCP Server Instructions；
2. 第一方 Skill；
3. Agent-specific rules，例如 `AGENTS.md`、Claude Rules、Cursor Rules；
4. 支持 Hook 的 Agent 使用 pre-tool/post-edit/stop hooks；
5. 可选 Git pre-commit / pre-push Gate；
6. Daemon watcher 产生 pending checkpoint，在下一次调用中强制显示；
7. GitHub Check 作为最终外部门禁。

## 13.8 第一方 Skills

```text
skills/
├── archcontext-develop/
├── archcontext-intervene/
├── archcontext-review/
└── archcontext-adr/
```

Skill 只描述调用顺序和用户沟通方式，不包含：

- Schema 规则；
- 写文件逻辑；
- Pressure 算法；
- Policy；
- Review 判断；
- GitHub API。

外部 Skill 可以增强表达风格，但不能决定产品核心状态。

# 14. ChatGPT App / GPT App 集成

## 14.1 目标

让用户能够在 ChatGPT 中：

- 讨论产品演进与系统架构；
- 查询当前项目的架构状态；
- 查看 Architecture Pressure；
- 审阅 Intervention、迁移方案和 Compatibility Debt；
- 让 ChatGPT 基于本地 Repository 的真实上下文提出方案；
- 将确认后的方案交给本地 Coding Agent 实施。

ChatGPT App 是架构讨论和决策 Surface，不取代本地 Coding Runtime。

## 14.2 关键技术约束

ChatGPT 不直接连接普通本地 stdio MCP。完整私有 Repository 能力需要：

```text
ChatGPT
  -> Secure MCP Tunnel
  -> local Streamable HTTP MCP
  -> archctxd
```

OpenAI Secure MCP Tunnel 使用用户机器发起的 outbound-only HTTPS 连接，不需要开放入站端口。但通过该通道返回给 ChatGPT 的内容仍会被发送至 OpenAI。产品承诺必须区分：

```text
成立：Repository 内容不经过或存储于 ArchContext SaaS。
不成立：使用 ChatGPT 本地隧道时，所有数据都永不离开用户机器。
```

## 14.3 双 MCP 通道

### 通道 A：Local Runtime MCP

```text
Transport: stdio / local Streamable HTTP
Execution: 用户本地
Repository access: 有，受工具策略控制
Primary clients: Codex、Claude Code、Cursor、ChatGPT Tunnel
```

### 通道 B：Cloud Metadata MCP

```text
Endpoint: https://api.archcontext.dev/mcp
Transport: Streamable HTTP
Execution: Cloudflare Worker
Repository content: 无
Primary client: ChatGPT App / account management
```

Cloud Metadata MCP 只提供：

- 账号状态；
- 订阅状态；
- GitHub Installation；
- Review Challenge / Check 状态；
- Device 与 Local Link 状态；
- Billing Portal；
- Public documentation。

它不得提供或代理：

- Task Context；
- Architecture Model；
- CodeGraph；
- 源码或 Diff；
- Detailed Finding；
- 本地 ChangeSet。

## 14.4 ChatGPT 工具分级

### 默认 Read / Plan 工具

|---|---|---|---|

## 14.6 数据分级与结果最小化

| 级别 | 示例 | Cloud MCP | Local Tunnel to ChatGPT |
|---|---|---:|---:|
| D0 Public | 文档、公开仓状态 | 允许 | 允许 |

## 14.7 OAuth 与身份

Cloud Metadata MCP 和发布型 ChatGPT App 需要符合 MCP Authorization 要求的 OAuth 2.1：

- Authorization Code + PKCE S256；
- Protected Resource Metadata；
- OAuth/OIDC Discovery；
- 正确验证 issuer、audience/resource、expiry 和 scopes；
- 不允许 Token Passthrough；
- Scope 按 Tool 最小化。

建议 Scope：

```text
account.read
entitlement.read
github.installations.read
github.reviews.read
devices.read
devices.link
billing.manage
architecture.read        # 仅 local tunnel resource
architecture.plan        # 仅 local tunnel resource
architecture.write       # 默认不授予
```

MVP 不应从零手写完整 OAuth Server。应通过 ADR 选择经过验证、支持 MCP 所需发现与 PKCE 的身份提供商或成熟 OAuth 组件；D1 继续保存应用数据而非承担全部协议安全。

## 14.8 连接体验

```bash
archctx gpt connect
```

流程：

1. 检查用户登录和 Pro/Public Entitlement；
2. 启动 local HTTP MCP，绑定 loopback；
3. 生成一次性 Local Link Token；
4. 启动或引导 Secure MCP Tunnel；
5. 显示 ChatGPT Connector 配置步骤；
6. 在本地显示当前暴露工具和数据策略；
7. 支持一键断开并撤销 Token。

### MVP

- ChatGPT Developer Mode / Custom App；
- Secure MCP Tunnel；
- 本地 Read/Plan 工具；
- 简单 Architecture Health UI。

### v1.1

在没有标准化的用户本地 Runtime 绑定流程前，不承诺将完整私有 Repository 能力作为无需本地设置的公共 App Directory 功能。

# 15. GitHub App 与本地签名 Review

## 15.1 GitHub App 职责

GitHub App 是 Repository Governance Plane：

- 安装与 Repository 选择；
- 接收 Pull Request 事件；
- 创建 Review Challenge；
- 创建和更新 Check Run；
- 验证 Attestation 对应当前 HEAD；
- 显示通过、失败、过期和等待状态。

GitHub App 不负责：

- Clone Repository；
- 读取 Contents；
- 运行 CodeGraph；
- 执行 LLM；
- 写入架构模型；
- 保存 Detailed Findings。

## 15.2 最小权限

```text
Repository permissions
├── Metadata: Read
├── Pull requests: Read
└── Checks: Write
```

默认不申请：

```text
Contents: Read
Contents: Write
Issues: Write
Actions: Write
Administration: Write
```

Webhook：

```text
installation
installation_repositories
pull_request: opened, reopened, synchronize, closed
check_run: rerequested
```

## 15.3 Review Challenge 状态机

```text
CREATED
QUEUED
CLAIMED
ATTESTED
VERIFIED
CHECK_PUBLISHED
STALE
EXPIRED
REJECTED
```

Challenge 字段：

```text
challengeId
installationId
repositoryId
pullRequestNumber
headSha
nonce
policyVersion
createdAt
expiresAt
```

## 15.4 Review Sequence

1. GitHub `pull_request.opened` 或 `synchronize` 到达 Worker。
2. Worker 验证 Webhook Signature 与 Delivery ID。
3. D1 幂等创建 Challenge。
4. Queue 创建或更新 GitHub Check 为 `queued`。
5. 本地 Runtime 查询该用户可处理的 Challenge。
6. Runtime 验证本地 Remote URL、Repository Numeric ID 与 HEAD SHA。
7. 执行 `complete_task` 或独立 `github review`。
8. 生成 Attestation 并用 Device Key 签名。
9. SaaS 验证签名、Nonce、订阅、版本和 SHA。
10. Queue 更新 Check Run。
11. 新 Push 到达时旧 Challenge 与 Check 立即 Stale。

# 16. Cloudflare SaaS 控制平面

## 16.1 服务边界

Control Plane 只处理：

- 用户与 GitHub Identity；
- Stripe Customer / Subscription；
- Entitlement；
- Device 与 Public Key；
- GitHub Installation / Repository Numeric ID；
- Review Challenge；
- Attestation；
- Webhook Delivery 去重；
- 最小审计事件；
- Metadata-only MCP。

## 16.2 Worker 路由

```text
/auth/*
/oauth/*
/.well-known/*
/api/account
/api/entitlement
/api/devices
/api/github/installations
/api/github/reviews
/api/attestations
/api/billing/checkout
/api/billing/portal
```

```text

/api/embeddings        prohibited
```

CI 应对生产路由清单执行 Privacy Contract Test。

## 16.3 为什么 D1 足够

该 SaaS 工作负载主要是按主键查询的低频元数据：

- Entitlement 每天或每 24 小时刷新；
- GitHub PR 产生少量 Challenge；
- 每次 Review 产生一个 Attestation；
- Webhook Delivery 短期保留；
- 不保存代码、索引、Embedding 或大文档。

D1 在 Workers Paid 中包含大量月度行读写额度，远高于 MVP 预计负载。需要关注的主要风险不是容量，而是无索引扫描和无节制 Telemetry。

## 16.4 D1 单库策略

MVP 使用一个 D1 Database：

```text
archcontext-production
```

适用原因：

- 数据量小；
- 查询短；
- 写入不密集；
- 事务边界简单；
- 运维成本最低。

分片触发条件：

- 单库写入排队成为可观测瓶颈；
- OAuth/Session 流量与 GitHub Attestation 相互影响；
- 单库接近 10 GB；
- 企业区域隔离要求。

可演进为：

```text
identity-db
billing-db
attestation-db-{shard}
```

## 16.5 Queue 使用

只使用一条 Queue：

```text
control-events
```

事件：

```text
github.check.create
github.check.update
stripe.entitlement.sync
webhook.retry
cleanup.expired-challenges
```

Queue Payload 必须小于 16 KB，理想小于 2 KB，只保存 ID 和版本，不保存原始 Webhook Body。

## 16.6 成本控制

- Entitlement JWT/签名令牌缓存 24 小时；
- 离线宽限 7 天；
- 不按 Tool Call 上报 Usage；
- Webhook Body 验证后立即丢弃；
- `webhook_deliveries` 保留 14 天；
- Challenge 在 PR 关闭后 7 天清理；
- Attestation 保留 90 天或用户可配置；
- 所有高频查询使用覆盖索引；
- Worker 设置单次 CPU 上限；
- 对 OAuth、Webhook、Attestation 分别限流；
- 日志采样，不记录 raw prompt 或 token。

## 16.7 可观测性

允许指标：

```text
request_count
status_code
route_id
latency_bucket
queue_retry_count
d1_rows_read
d1_rows_written
challenge_age
attestation_verification_result
```

禁止日志：

```text
source code
tool input/output containing repository context
raw GitHub webhook body
OAuth token
refresh token
Stripe full payload after processing
ChatGPT prompt text
```

# 17. 架构领域模型与 Schema

## 17.1 建模目标

Schema 必须同时满足：

- Agent 易于精确读取和更新；
- Git Diff 易审查；
- 可进行确定性校验和 Digest；
- 支持渐进式复杂度；
- 不绑定 C4 或单一可视化工具；
- 能表达目标状态、迁移和兼容债；
- 高频、易漂移的代码证据不污染 Git 历史。

## 17.2 Source of Truth 分层

```text
Source Code
= 实现事实

CodeGraph
= 可查询的代码事实索引

.archcontext/model + decisions + policies
= 声明架构与工程意图的 Git 事实源

ArchContext SQLite
= Observed Evidence、Verification、Task、Review 与 Cache

.generated
= 面向人的可重建投影

SaaS D1
= 身份、计费、GitHub 元数据与证明
```

## 17.3 核心实体

| 实体 | 是否提交 Git | 说明 |
|---|---:|---|
| Product | 是 | 产品目的、用户、成功标准和风险域 |
| Architecture Node | 是 | 模块、能力、接口、数据存储等稳定语义 |
| Architecture Relation | 是 | 声明的交互和依赖意图 |
| Constraint | 是 | 允许、禁止和必须满足的规则 |
| ADR | 是 | 重要决策与理由 |
| Compatibility Contract | 是 | 兼容代码的合法依据与移除机制 |
| Architecture Intervention | 是 | 结构性改变的目标、迁移与验证 |
| Policy | 是 | Review、迁移和写入策略 |
| Observed Evidence | 否 | 文件、Symbol、行号、Commit、Confidence |
| Verification State | 否 | Declared 与 Observed 的对齐结果 |
| Task Session | 否 | 当前 Agentic SOP 状态 |
| ChangeSet Draft | 默认否 | 可选保存为审计文档 |
| Review Result Detail | 否 | 保留本地；只上传 Digest |

## 17.4 Schema 共同规则

- `schemaVersion` 必填；格式为 `archcontext.<entity>/v1`。
- ID 使用稳定的小写 dot notation，不随显示名称改变。
- 时间使用 UTC ISO 8601。
- 核心 Schema 默认 `additionalProperties: false`。
- 扩展字段统一放入 `extensions`，键以反向域名或厂商前缀命名。
- 路径统一为 Repository-relative POSIX 格式。
- YAML 写入使用 UTF-8、LF、两空格缩进、稳定键排序。
- Digest 基于规范化 JSON 表示计算，而非原始 YAML 字节。
- 所有引用必须通过 ID，禁止以文件路径作为实体主键。
- 删除实体前必须验证反向引用。

## 17.7 Architecture Node Schema

### 支持的 `kind`

MVP：

```text
capability
module
component
interface
event
datastore
external-system
deployment
```

`service` 不作为强制独立 Kind；可由 `module` 加 `deployable: true` 表达，避免简单应用被迫采用微服务术语。

### 示例

```yaml
schemaVersion: archcontext.node/v1
id: module.subscription
kind: module
name: Subscription
status: active
parent: capability.subscriptions

summary: Owns the subscription lifecycle and plan transitions.

responsibilities:
  - create-subscription
  - change-plan
  - cancel-subscription
  - expose-subscription-status

source:
  include:
    - src/subscription/**
  exclude:
    - src/subscription/generated/**
  entrypoints:
    - src/subscription/index.ts

ownership:
  lifecycle:
    - subscription
  data:
    - datastore.subscription

interfaces:
  provides:
    - interface.subscription-api
    - event.subscription-changed
  consumes:
    - interface.payment-api

criticality: high
riskDomains:
  - payments

notes:
  - Payment credentials must never be stored in this module.
```

### 必要校验

- `module` 必须有至少一个 responsibility；
- source include 不得落入另一个 active module 的 exclusive selector，除非显式 shared；
- datastore 只能有一个 primary owner；
- public interface 必须有 owning node；
- status=removed 的节点不能被 active relation 引用。

## 17.8 Architecture Relation Schema

```yaml
schemaVersion: archcontext.relation/v1
id: relation.subscription-calls-payment
kind: calls
source: module.subscription
target: module.payment

via: interface.payment-api
protocol: https
synchrony: synchronous

intent: Authorize or refund payment through the payment boundary.

allowedData:
  - payment-reference
  - amount
  - currency

forbiddenData:
  - card-number
  - payment-credential

criticality: high
```

支持的 `kind`：

```text
depends-on
calls
publishes
subscribes
reads
writes
implements
deployed-on
```

必要校验：

- `reads/writes` 的 target 必须为 datastore；
- `publishes/subscribes` 必须通过 event；
- `calls` 指向跨模块目标时建议指定 interface；
- 违反 Constraint 的 relation 无法通过 Review；
- 与 Observed Edge 不一致时进入 DRIFT，而非自动覆盖声明。

## 17.9 Constraint Schema

```yaml
schemaVersion: archcontext.constraint/v1

severity: error

  nodes:
    - module.subscription

  type: forbid-data-access
  target: datastore.payment

    - reads
```


## 17.16 Digest 规范


文件名不是实体 ID 的事实源，但必须与 ID 可验证对应。

## 18.4 Generated 文件策略

- 默认提交 Git，便于 GitHub 浏览；
- 内容顶部标记“Generated by ArchContext”；
- 通过 modelDigest 判断是否 stale；
- 手工修改会在 `validate` 中失败；
- 生成过程稳定排序，避免无意义 Diff；
- 可配置 `commitToGit: false`，但公开项目建议提交。

## 18.5 本地状态目录

不在 Repository 内保存 ArchContext SQLite，避免误提交和分支冲突。

```text
macOS:
~/Library/Application Support/ArchContext/repos/<fingerprint>/

Linux:
~/.local/share/archcontext/repos/<fingerprint>/

Windows:
%LOCALAPPDATA%\ArchContext\repos\<fingerprint>\
```

内容：

```text

state.db-shm
workspace.json
locks/
cache/
snapshots/
logs/
```

CodeGraph 自己的 `.codegraph/` 由 CodeGraph 管理，ArchContext 不修改其内部文件。

## 18.6 Agent 配置写入

安装器支持：

```text
Codex: AGENTS.md / MCP config
Claude Code: CLAUDE.md / MCP config / hooks
Cursor: .cursor/rules / MCP config
Generic: printed stdio MCP JSON
```

所有配置修改必须：

- 使用可识别 Marker；
- 可重复执行而不重复追加；
- 支持 `archctx uninstall` 精确撤销；
- 不覆盖用户已有规则；
- 将核心 SOP 保持简短，详细规则由 MCP Instructions 提供。

# 19. 本地 SQLite Schema

## 19.1 设计目标

本地 SQLite 保存高频、可重建、与具体工作区相关的派生状态。它不是团队协作事实源，也不替代 CodeGraph DB。

必须支持：

- 单 Writer、多 Reader；
- 多 Agent 并发查询；
- 崩溃恢复；
- FTS5 文档检索；
- Snapshot 与 Stale 检测；
- 不复制完整 CodeGraph；
- 可通过 `archctx rebuild` 从 Git 与 CodeGraph 重建。

## 19.2 SQLite 配置

```sql
PRAGMA journal_mode = WAL;
PRAGMA foreign_keys = ON;
PRAGMA synchronous = NORMAL;
PRAGMA busy_timeout = 5000;
PRAGMA temp_store = MEMORY;
```

安全要求：

- 数据库文件权限仅当前用户可读写；
- 不保存 OAuth Access Token；
- Device Private Key 不进入 SQLite；
- Source snippet cache 可配置关闭，并默认短期 TTL；
- 日志和 DB 不保存 Secret。

# 21. 安全、隐私与信任模型

![信任边界与数据流](archcontext_prd_v2_assets/trust_boundaries_portrait.png)

## 21.1 隐私承诺

对外准确表述：

> **ArchContext SaaS 永不接收或存储你的源代码、Diff、架构模型、CodeGraph 或详细 Review 结果。**

不得笼统表述：

> “所有数据永不离开本地。”

原因：当用户主动启用 ChatGPT App、其他远程 Agent、GitHub 评论或外部模型时，用户选择的数据可能发送给对应供应商。ArchContext 必须准确说明每条数据路径。

## 21.2 信任区域

### Zone A：用户本地

最高敏感：

- Source Code；
- Git Diff；
- Secret；
- CodeGraph；
- Architecture Model；
- ADR 和 Policy；
- Detailed Findings；
- Local SQLite；
- Device Private Key。

### Zone B：用户选择的 Agent / OpenAI

当用户使用 ChatGPT Secure MCP Tunnel 或远程 Agent 时，任务级 Tool Input/Output 可进入该 Agent 提供商。ArchContext 负责最小化和脱敏，但不能将其描述为本地数据。

### Zone C：ArchContext SaaS

只允许：

- Identity；
- Billing；
- Entitlement；
- Device Public Key；
- GitHub Numeric Metadata；
- SHA；
- Digest；
- Pass/Fail；
- Runtime Version。

### Zone D：GitHub / Stripe

按照其各自 API 和权限处理 Repository 元数据、Check 或计费状态。

## 21.3 威胁模型

| 威胁 | 场景 | 缓解 |
|---|---|---|
| Prompt Injection | Repository 文档诱导 Agent 泄露 Secret 或绕过 Policy | 将 Repo 内容视为不可信；Core 强制权限；Secret Redaction；Tool 不能靠 Prompt 放权 |
| Path Traversal | ChangeSet 写入 Repository 外部 | canonical path、allowlist、symlink resolution、openat/安全文件 API |
| Symlink Escape | `.archcontext` 内符号链接指向外部 | 写前 realpath；默认拒绝 symlink target |
| Stale Plan | Agent 基于旧 Worktree 套用变更 | expectedHash、HEAD、Worktree Digest、Session Version |
| Attestation Replay | 重放旧签名通过新 PR | Challenge Nonce、head SHA、过期时间、唯一约束 |
| Device Key Theft | 本地私钥被复制 | OS Keychain、可撤销 Key、设备列表、短期 Challenge |
| Webhook Forgery | 伪造 GitHub/Stripe 事件 | 签名验证、时间窗口、Delivery ID 去重 |
| OAuth Token Leakage | Token 出现在日志或 Tool Result | Keychain、redaction、never echo、短 TTL、scope |
| Tunnel Overexposure | 本地 HTTP MCP 被局域网访问 | loopback only、ephemeral port/token、自动关闭 |
| Secret Exfiltration to GPT | Agent 要求完整文件并包含 Secret | secret scanner、deny patterns、explicit confirmation、size cap |
| Dependency Compromise | CodeGraph 或 npm 依赖被攻击 | exact pin、lockfile、provenance/checksum、SBOM、release review |
| Malicious Architecture File | YAML alias bomb、超大文件、复杂 regex | size/depth limits、安全 parser、glob/regex 限制 |
| Denial of Wallet | Cloud API 无限请求或无索引扫描 | rate limit、CPU cap、query metrics、budget alarms |
| Local DoS | 巨型 Repo 导致持续 Index/Review | cancellation、resource limits、incremental sync、backpressure |

## 21.4 Tool 权限模型

工具按能力分类：

```text
READ_MODEL
READ_CODE_FACTS
READ_SOURCE_SNIPPET
PLAN_CHANGE
WRITE_ARCHITECTURE
RUN_REVIEW

```

每个 Channel 绑定 Capabilities，Agent Tool Annotation 只帮助交互，不替代服务端授权。

## 21.5 Secret 防护

Local Gateway 在返回远程 Agent 前执行：

1. 路径敏感度过滤；
2. 常见 Secret Pattern；
3. 高熵 Token 检测；
4. `.env`、Credential、Key 文件默认禁止；
5. 结构化字段 allowlist；
6. 结果长度限制；
7. 必要时要求本地确认。

永久拒绝返回：

```text
private keys
access tokens
refresh tokens
passwords
raw environment variables
cloud credentials
GitHub installation tokens
Stripe secrets
```

## 21.6 文件写入安全

- 所有写入只允许 `.archcontext/`，除非特定 Operation 明确允许更新 Agent Rule Marker；
- 不使用 shell 拼接文件路径；
- 拒绝绝对路径与 `..`；
- 写入前后均验证 Repository Root；
- 使用 lock file 防止并发 Apply；
- 生成文件 Header 含 Model Digest；
- Apply 失败恢复备份；
- 所有 YAML 经过 Schema 和跨实体引用校验。

## 21.7 CodeGraph 供应链

- 精确锁版本和 package integrity；
- Adapter Contract Test 覆盖所有依赖方法；
- Release 升级由人工批准 ADR/PR；
- 不自动安装 `latest` 用于可签名 Review；
- `archctx doctor` 显示验证状态；
- 不访问 CodeGraph DB；
- 不将 CodeGraph Telemetry 默认继承为开启状态；
- 安装时明确列出 CodeGraph License 和版本。

## 21.8 OAuth 与 Token

### CLI / Local Runtime

- 浏览器 Authorization Code + PKCE 或 Device Flow；
- Refresh Token 存 OS Keychain；
- Access Token 短期有效；
- Entitlement 可使用签名离线令牌；
- Token 不进入 Repository 配置或 SQLite。

### GitHub App

- Installation Token 按需获取；
- 只在 Worker 内存使用；
- 不发送给 Local Runtime；
- 不记录到日志；
- 使用最小 Repository 权限。

### ChatGPT App

- OAuth 2.1；
- Scope 与 Tool 匹配；
- aud/resource 验证；
- PKCE S256；
- Token 不透传到下游 GitHub 或 Stripe。

## 21.9 安全测试门禁

## 22.3 日常开发 SOP

用户：

> “增加团队邀请与角色权限。”

Agent 自动调用：

```text
archcontext_prepare_task
```

返回：

```text
Posture: Structural
Relevant: auth, team, invitation
Constraint: role checks must stay in authorization module
Risk: invitation acceptance currently bypasses central permission check
Required checkpoint: after data model change
```

Agent 实现后调用：

```text
archcontext_checkpoint
```

若检测到新旧权限路径并存：

```text
Architecture Pressure: High
Posture change: Intervention
Reason: Two authorization owners; new fallback duplicates role evaluation
Recommendation: Move all role evaluation to module.authorization
```

用户只需决定产品级问题，例如：

> “邀请接受后，成员权限应立即生效，还是需管理员再次确认？”

## 22.4 必要重构流程

1. Runtime 检测高结构压力；
2. CodeGraph 盘点调用方与影响；
3. Refactor Confidence 不足则生成 Proof Point；
4. Agent 完成最小端到端验证；
5. Runtime 生成 Intervention；
6. 用户批准目标状态；
7. Agent 实现阶段迁移；
8. 每次 Checkpoint 更新 Migration Progress；
9. Cleanup Gate 删除旧路径；
10. Intervention 标记 completed。

## 22.5 Compatibility 流程

当 Agent 尝试新增 Adapter：

```text
Agent: 为避免破坏旧调用，我会保留 LegacyMapper。

ArchContext:
No real compatibility constraint found.
- All 7 callers are internal and tracked by CodeGraph.
- No persisted data or external contract depends on the mapper.
Recommended: migrate callers and delete LegacyMapper.
```

若确有外部消费者：

```text
ArchContext creates Compatibility Contract
-> owner
-> consumers
-> expiresAt
-> removal conditions
-> migration issue
```

# 25. Evals、指标与验收体系

## 25.1 北极星指标

> **长期项目中，Agent 能否在不增加结构性技术债的情况下持续完成新需求。**

代理指标：

- 任务首次准备成功率；
- 后续 Session 复用历史决策率；
- 不合理 Compatibility Code 引入率；
- 结构问题在 Commit 前发现率；
- Intervention 完成后的复杂度下降；
- 用户被迫回答技术术语问题的数量；
- Repository 增长后 Task Completion Time 变化；
- 任务完成后遗留临时路径数量。

## 25.2 核心 Eval 套件

### Eval A：Patch vs Refactor

构造同一需求的长期演进 Repository：

- Agent 无 ArchContext；
- Agent + CodeGraph；
- Agent + CodeGraph + ArchContext。

评估：

- wrapper 数量；
- 分支复杂度；
- 重复概念；
- 变更半径；
- 旧路径是否删除；
- 后续任务成功率。

### Eval B：Compatibility Debt

场景：内部调用方很多，但无外部契约。  
成功标准：ArchContext 推荐迁移调用方，而非永久保留 shim。

场景：确有外部 Webhook 消费者。  
成功标准：生成 Compatibility Contract，而非直接删除。

### Eval C：High Pressure / Low Confidence

场景：疑似架构问题但调用方覆盖不足。  
成功标准：进入 Proof Required，不选择补丁或无脑重写。

### Eval D：Target vs Migration

场景：需要阶段双写。  
成功标准：双写被标记为 Migration State，有完成条件；后续任务会推动清理。

### Eval E：Context Budget

评估 Task Context 在 8/12/20 KB 下：

- 约束召回；
- 相关节点召回；
- Agent 成功率；
- Tool Call 数；
- 无关内容比例。

### Eval F：Long Horizon

同一 Repository 连续执行 20-50 个需求，评估：

- 架构模型准确度；
- 技术债趋势；
- Agent 重复理解成本；
- 历史决策引用；
- 最终可维护性。

## 25.3 质量指标

| 指标 | MVP 目标 |
|---|---:|
| Schema Validation Precision | 100% deterministic |
| Stale Plan 拦截率 | 100% |
| Path Escape 拦截率 | 100% |
| Unjustified Compatibility 检测 Recall | >= 85% on eval set |
| Architecture Drift Precision | >= 90% on supported patterns |
| Context Constraint Recall | >= 95% |
| Context 无关内容比例 | <= 20% |
| ChangeSet 原子恢复成功 | 100% fault-injection tests |
| Attestation Replay 拦截 | 100% |
| SaaS Code-content Route | 0 |

# 27. 风险与缓解

## 27.1 产品风险

| 风险 | 影响 | 概率 | 缓解策略 |
|---|---|---:|---|
| 用户把产品误解为架构文档生成器 | 无法理解主动控制价值 | 高 | 首页、Onboarding 和 Demo 以“任务前/中/后 SOP”展示，不以 Diagram 为核心 |
| Agent 忽略 MCP SOP | 架构循环未被执行 | 高 | Skill + Hook + Complete Task Gate；关键任务状态持久化；对支持的 Host 提供自动配置 |
| 系统过度触发重构 | 用户失去信任 | 中 | Pressure 与 Confidence 分离；高压力低信心必须 Proof Required；展示证据和可逆路径 |
| 系统仍然过度保守 | 不能解决核心痛点 | 中 | Eval 强制比较 Patch 与 Target State；兼容层持证上岗；重复热点触发 Intervention |
| 非专家无法做架构决定 | 流程卡住 | 高 | 将问题翻译为产品责任、数据风险和成本；推荐默认方案；技术细节渐进展示 |
| 简单项目被过度治理 | 破坏 Vibe Coding 速度 | 高 | Progressive Architecture；L0 最小模型；只在复杂度信号出现时升级 |
| $5 ARPU 支撑不了高支持成本 | 商业不可持续 | 中 | 本地计算、自助 Onboarding、公开诊断、无托管索引；团队/企业能力后置独立定价 |

## 27.2 技术风险

| 风险 | 影响 | 缓解策略 |
|---|---|---|
| CodeGraph API 或行为快速变化 | 核心功能中断 | 精确锁版本；Adapter Contract；Fixture；Compatibility Matrix；禁止内部 DB 耦合 |
| CodeGraph Evidence 不完整 | 错误判断重构安全性 | Confidence 下降；Proof Required；不把 Heuristic 冒充 Verified Fact |
| 大型 Repository 索引慢 | Context 延迟过高 | Daemon、增量更新、Blob Hash、Changed Scope、后台预热、预算式查询 |
| SQLite 多进程写冲突 | Runtime 不稳定 | 单 Writer Daemon；WAL；Busy Timeout；CLI/MCP 不直接持有 Writer 连接 |
| 架构模型与代码持续漂移 | 上下文污染 | Checkpoint、Review、Verified State、Staleness TTL、Commit Binding |
| Agent 生成错误 ChangeSet | 模型损坏 | Structured Output、Schema、Policy、Precondition、Preview、Atomic Apply、Rollback |
| Schema 变化破坏用户 Repository | 信任损失 | 版本化迁移、备份、Dry Run、可逆迁移、旧版本 Reader 兼容窗口 |
| MCP Tool 数量膨胀 | Tool Selection 与 Context 成本增加 | 五个 Workflow Tool 上限；对象走 Resources；高级能力通过 CLI |
| 本地 Runtime 被多个 Host 并发调用 | 状态竞争 | Session Lease、Operation ID、事务、Repository-level Write Lock |

## 27.3 安全与隐私风险

| 风险 | 影响 | 缓解策略 |
|---|---|---|
| SaaS 无意记录 Repository 内容 | 违反核心承诺 | API Contract 禁止；Body Logging 关闭；字段 Allowlist；日志 Redaction；集成测试扫描 |
| ChatGPT Tool 返回私有内容 | 用户误解“本地”含义 | 首次启用 Disclosure；每工具 Data Class；敏感 Scope 默认关；UI 明示目的地 |
| Secure MCP Tunnel 被滥用 | 私有 Runtime 暴露 | Outbound-only；短期 Session；设备绑定；Scope；撤销；Loopback 默认 |
| Prompt Injection 诱导危险 Tool | 文件被误改 | Read-only 默认；写 Tool 不在默认 GPT Surface；本地审批；Allowed Path；Post-validation |
| GitHub Webhook 重放 | Check 状态被污染 | HMAC、Delivery ID、TTL、幂等表、最小 Payload |
| Attestation 伪造或重放 | 错误 PR 通过 | Device Key、Challenge Nonce、Repository/PR/SHA Binding、单次消费、时钟窗口 |
| 本地开发者可篡改客户端 | Attestation 信任被高估 | 明确称 Developer-attested；企业强保证使用客户受控 Runner |
| OAuth Token 泄漏 | 账户被接管 | PKCE、短期 Token、Keychain、Scope 最小化、Rotation、Device Revocation |

## 27.4 生态风险

| 风险 | 影响 | 缓解策略 |
|---|---|---|
| Agent Host 改变 MCP/Skill 行为 | 集成失效 | 标准 MCP 优先；Host Adapter 薄；CI 兼容测试；不把核心逻辑放 Skill |
| OpenAI / ChatGPT App 规范变化 | GPT App 能力中断 | MCP Apps 标准优先；ChatGPT 扩展 Feature Detect；无 UI 仍可工作 |
| 用户已使用 LikeC4/Structurizr | 不愿迁移 | 后置 Import/Projection Adapter；Native Schema 只承载 Agent Runtime 所需语义 |
| 竞争平台加入架构记忆 | 差异缩小 | 聚焦 Evidence、Intervention、Compatibility Debt、Attestation 和跨 Agent 中立性 |

## 27.5 风险触发器

以下指标触发架构或产品复审：

- CodeGraph Adapter 相关失败占任务失败超过 5%；
- `prepare_task` P95 超过 5 秒；
- Intervention 被用户拒绝超过 70%；
- Unjustified Compatibility 误报超过 15%；
- ChangeSet 回滚率超过 3%；
- 用户关闭 Complete Task Gate 超过 30%；
- ChatGPT Tunnel 导致的隐私支持请求持续增长；
- D1 或 Worker 单用户边际成本超过月收入的 10%；
- MCP Tool Call 数随 Repository 规模线性增长；
- Long Horizon Eval 中技术债指标持续恶化。

# 28. 架构决策记录（ADR）清单

以下 ADR 必须在实现前建立，并与本 PRD 保持一致：

| ADR | 标题 | 核心决策 |
|---|---|---|
| ADR-0001 | Agentic Architecture Control Loop | ArchContext 是任务 SOP，不是被动文档查询器 |
| ADR-0002 | CodeGraph as Required Code Facts Engine | CodeGraph 是产品硬依赖，但仅经 Adapter 访问 |
| ADR-0003 | Local-first Trust Boundary | Repository 内容不进入 ArchContext SaaS |
| ADR-0004 | SQLite Local Store | 本地使用 SQLite，不使用 PGlite |
| ADR-0005 | Single-writer Runtime Daemon | `archctxd` 管理 Session、DB 与 CodeGraph 生命周期 |
| ADR-0006 | CLI and MCP as Thin Adapters | CLI/MCP 调用共享 Core，不复制业务逻辑 |
| ADR-0007 | Structured Architecture Source of Truth | `.archcontext/model` 为结构化事实源；Markdown 为决策或投影 |
| ADR-0008 | Declared / Observed / Verified | 意图、代码事实与验证状态分层 |
| ADR-0009 | Target State vs Migration State | 目标架构和临时迁移严格分离 |
| ADR-0010 | Compatibility Code Requires Contract | 兼容层必须有真实契约、Owner 和移除条件 |
| ADR-0011 | Architecture Intervention | 结构性重构作为一等领域对象 |
| ADR-0012 | ChangeSet-only Architecture Writes | Agent 不直接写模型；所有写入经 ChangeSet |
| ADR-0013 | Progressive Architecture | 治理复杂度随项目复杂度升级 |
| ADR-0014 | Context Compiler with Budget | Agent 使用任务级、预算化上下文，不加载全部模型 |
| ADR-0015 | GitHub App without Contents Permission | 默认 GitHub App 无代码读取权限 |
| ADR-0016 | Signed Local Attestation | Review 本地执行，SaaS 验证最小证明并发布 Check |
| ADR-0017 | Cloudflare Control Plane | Workers + D1 + Queue 承载身份、计费和元数据 |
| ADR-0018 | Dual MCP Surface | 本地完整 MCP 与 SaaS 元数据 MCP 分离 |
| ADR-0019 | ChatGPT via Secure MCP Tunnel | 私有 Runtime 通过显式、短期 Tunnel 暴露给 ChatGPT |
| ADR-0020 | MCP Apps Standard-first UI | UI 使用 `_meta.ui.resourceUri` 与 `ui/*`，ChatGPT 扩展仅增强 |

| ADR-0025 | Evidence Confidence and Proof Required | 证据不足时先证明，不默认打补丁或重写 |

# 附录 A：错误模型

所有可机器处理错误使用稳定 Code：

| Code | 含义 | 可重试 | Agent 动作 |
|---|---|---:|---|
| `AC_REPO_NOT_FOUND` | 未发现 Repository Root | 否 | 请求用户打开正确目录 |
| `AC_RUNTIME_UNAVAILABLE` | `archctxd` 不可用 | 是 | 启动或重连 Runtime |
| `AC_CODEGRAPH_UNAVAILABLE` | CodeGraph 未启动或不兼容 | 是 | 运行 Diagnostics，不做无证据架构结论 |
| `AC_INDEX_STALE` | CodeGraph Snapshot 过期 | 是 | 增量 Sync 后重试 |
| `AC_CONTEXT_STALE` | Task Context 与 HEAD/Worktree 不一致 | 是 | 重新 Prepare Task |
| `AC_SCHEMA_INVALID` | 架构模型不符合 Schema | 否 | 生成修复 Plan，不直接忽略 |
| `AC_POLICY_VIOLATION` | ChangeSet 违反策略 | 否 | 修改 Plan 或显式审批例外 |
| `AC_PATH_DENIED` | 写入路径不允许 | 否 | 禁止绕过 |
| `AC_PRECONDITION_FAILED` | 文件 Digest 或状态已变化 | 是 | 重建 Plan |
| `AC_INTERVENTION_REQUIRED` | 局部补丁不足 | 否 | 进入 Intervention SOP |
| `AC_PROOF_REQUIRED` | 重构证据不足 | 否 | 执行 Proof Point |
| `AC_COMPAT_CONTRACT_REQUIRED` | 新兼容路径无真实契约 | 否 | 删除路径或建立合同 |
| `AC_USER_CONFIRMATION_REQUIRED` | 高影响写入需确认 | 是 | 向用户显示可理解的决策 |
| `AC_ENTITLEMENT_REQUIRED` | 私有 Repo 无有效 Pro 权益 | 否 | 登录或订阅 |
| `AC_ATTESTATION_REJECTED` | Attestation 校验失败 | 否 | 显示原因并重新 Review |
| `AC_TUNNEL_SCOPE_DENIED` | GPT App 请求超出 Tunnel Scope | 否 | 缩小请求或本地授权 |

错误响应：

```json
{
  "schemaVersion": "1",
  "ok": false,

    "code": "AC_CONTEXT_STALE",
    "message": "Task context was prepared for a different worktree state.",

    "action": {
      "tool": "archcontext_prepare_task",
      "arguments": { "refresh": true }
    }
  },
  "snapshot": {
    "expectedHead": "abc123",
    "actualHead": "def456"
  }
}
```

# 附录 B：MCP Tool Contract 摘要

## B.1 `archcontext_prepare_task`

**目的：** 在 Coding 前编译最小但完整的任务级架构上下文。

输入：

```json
{
  "task": "增加团队成员邀请与角色权限",
  "intent": "implement",
  "budget": {
    "maxBytes": 16000,
    "maxEvidence": 20
  }
}
```

输出重点：

```text
posture
related capabilities / nodes
constraints
decisions
source evidence
architecture pressure
refactor confidence
target or proof recommendation
required checkpoints
```

## B.2 `archcontext_checkpoint`

**目的：** 在 Agent 完成一个结构性步骤后重新观察变化。

输入包含 Task ID、当前阶段和可选 Changed Files。输出可能要求：

```text
continue
re-plan
create intervention
create compatibility contract
request user decision
```

## B.3 `archcontext_plan_update`

**目的：** 生成结构化 ChangeSet，不直接写文件。

必须包含：

- 操作列表；
- Before/After Digest；
- 影响摘要；
- Policy Result；
- 用户可理解解释；
- 是否需要确认；
- Rollback 信息。

## B.4 `archcontext_apply_update`

**目的：** 应用已批准 ChangeSet。

安全要求：

- ChangeSet 必须处于 Approved；
- Snapshot 未过期；
- Path 允许；
- 文件 Precondition 成立；
- 原子写入；
- Apply 后验证；
- 失败时回滚。

本工具默认不出现在 ChatGPT App 的公共 Tool Surface。

## B.5 `archcontext_complete_task`

**目的：** 任务完成前的 Architecture Gate。

输出：

```text
pass
pass_with_followups
blocked
```

阻塞条件至少包括：

- 未解决高严重度 Policy Violation；
- 未同步的架构变化；
- 无依据兼容层；
- 超期 Migration State；
- Task Context 已过期；
- Review 未绑定当前 Worktree。

# 附录 C：发布验收场景

## C.1 从简单 Web 到订阅产品

1. 初始化一个单体 Web 应用。
2. 添加登录、订阅、支付 Webhook 和后台任务。
3. ArchContext 从 L0 升级为 L1。
4. Agent 获得支付数据边界和 Lifecycle Owner。
5. Agent 不得把付款凭据写入业务数据库。
6. 架构模型、ADR 和 Generated View 同步。

## C.2 阻止永久兼容层

1. 任务要求统一旧、新状态字段。
2. CodeGraph 显示旧字段只有内部消费者。
3. Agent 建议添加永久 Mapper。
4. ArchContext 拒绝无真实契约的兼容层。
5. 生成调用方迁移与旧字段删除 Kill List。
6. 完成后 Repository 只保留单一路径。

## C.3 高风险重构先证明

1. Architecture Pressure 高。
2. CodeGraph 无法确认两个动态调用方。
3. Refactor Confidence 低。
4. Runtime 进入 Proof Required。
5. Agent 建立最小探针、测试或运行时日志以确认调用。
6. 证据完成后再决定重构或兼容。

## C.4 ChatGPT App Read-only

1. 用户启动本地 Runtime 与 Secure MCP Tunnel。
2. ChatGPT 调用 Task Context。
3. UI 显示 Repository、HEAD 和将发送的数据类别。
4. ArchContext Cloud 不收到 Repository 内容。
5. ChatGPT 不能调用 Apply ChangeSet。
6. 用户撤销 Tunnel 后调用立即失败。

## C.5 GitHub Signed Review

1. GitHub App 收到 PR synchronize。
2. 创建绑定 Head SHA 的 Challenge 与 queued Check。
3. 本地 Runtime Review 当前 Worktree。
4. 生成 Device-signed Attestation。
5. SaaS 验证最小字段并更新 Check。
6. PR Push 新 Commit 后旧 Attestation 自动失效。
